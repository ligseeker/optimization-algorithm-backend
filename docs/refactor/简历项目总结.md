# 简历项目总结

## 项目名称
流程优化任务管理平台（optimization-algorithm-backend）

## 项目背景
原项目以 YAML/JSON 文件作为主数据源，存在数据一致性、权限控制、并发扩展和任务追踪困难等问题。项目通过渐进式重构，保留算法核心能力，完成向 MySQL + Redis + Sa-Token + MyBatis-Plus 架构迁移。

## 技术栈
- Java 11
- Spring Boot 2.7.14
- MyBatis-Plus
- MySQL 8
- Redis
- Sa-Token
- ThreadPoolTaskExecutor
- Springdoc OpenAPI
- JUnit 5 / Mockito

## 核心功能
- 用户认证与权限控制（ADMIN/USER）
- 工作空间与流程图（节点/路径/装备/约束）结构化 CRUD
- YAML 导入导出（仅作为交换格式）
- 优化任务提交、状态查询、失败重试、结果查询
- 展示态流程图版本化缓存（graph_version）
- 操作日志 AOP（成功/失败统一记录）

## 个人职责
- 设计并落地统一分层：Controller / Service / Mapper / Converter / DTO / VO
- 完成核心表结构与 MyBatis-Plus 持久化层建设
- 实现 Sa-Token 登录、鉴权与资源访问校验
- 实现流程图结构化 CRUD 与事务边界
- 实现 YAML 导入导出及结构化校验错误报告
- 完成任务中心异步执行与状态机收敛
- 设计并实现 Redis 缓存策略、操作日志与接口文档
- 补齐关键自动化测试并完善项目文档

## 项目亮点
- 渐进式重构：不中断算法主流程，兼容旧能力并可回滚
- 缓存一致性方案：编辑态只改 MySQL，展示态按 `graph:detail:{graphId}:v{graphVersion}` 版本化缓存
- 任务链路可观测：PENDING/RUNNING/SUCCESS/FAILED 全链路状态缓存与回源兜底
- 安全与审计：统一鉴权、统一异常、操作日志可追踪且脱敏

## 面试可讲问题
- 为什么 YAML 只能做导入导出，不能做主存储
- 为什么流程图缓存采用版本化而不是局部更新
- 如何保证 Redis 异常不影响主业务
- 如何在不改算法核心的前提下接入任务中心
- USER/ADMIN 资源隔离如何实现并验证
- 重试策略为何建议“新任务而非覆盖旧任务”

## 可继续扩展方向
- 引入 Flyway/Liquibase 管理数据库版本迁移
- 增强任务执行隔离与限流（线程池分级、配额控制）
- 增加更细粒度权限模型（按 workspace/graph 授权）
- 增加 graph_diff 与版本对比可视化
- 对接前端下载流式导出与大文件导入异步化
