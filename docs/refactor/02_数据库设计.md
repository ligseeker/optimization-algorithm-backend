# 02_数据库设计

## 1. 结论

基于 [00_重构总说明.md](C:/Users/zhangll/Desktop/optimization-algorithm-backend/docs/refactor/00_重构总说明.md) 和 [01_项目现状分析.md](C:/Users/zhangll/Desktop/optimization-algorithm-backend/docs/refactor/01_项目现状分析.md)，现有数据库设计文档需要调整，原因如下：

- 需要严格收敛到第一版限定表范围；
- 需要显式支撑工作空间、异步优化任务、简化版 Diff、YAML 导入导出；
- 需要避免把算法模型直接映射为 Entity；
- 需要为 MyBatis-Plus、Redis、Sa-Token、ThreadPoolTaskExecutor 预留清晰边界。

本版数据库第一期只设计以下 10 张表：

- `sys_user`
- `workspace`
- `flow_graph`
- `equipment`
- `process_node`
- `process_path`
- `constraint_condition`
- `optimize_task`
- `optimize_result`
- `operation_log`

暂不设计：

- 复杂 RBAC
- 消息队列
- 完整 `graph_diff`
- `flow_version`

## 2. 设计原则

- `MySQL 8` 是最终数据源，`Redis` 只做缓存，不做事实存储。
- `ProcessMap / MultiNode / ProcessPath / Equipment / ConstraintCondition` 继续作为算法模型，不直接当数据库 Entity。
- 所有业务主表统一包含审计字段与逻辑删除字段，便于 MyBatis-Plus 统一处理。
- 所有资源都必须可以追溯到当前用户可访问的 `workspace`。
- 优化任务与优化结果分表，任务负责状态，结果负责结果快照与简化 Diff。
- YAML 只作为导入导出格式，不再作为主存储格式。

## 3. 表关系总览

```text
sys_user 1 ── N workspace
workspace 1 ── N flow_graph
flow_graph 1 ── N equipment
flow_graph 1 ── N process_node
flow_graph 1 ── N process_path
flow_graph 1 ── N constraint_condition
flow_graph 1 ── N optimize_task
optimize_task 1 ── 1 optimize_result

operation_log 独立记录用户操作，可关联 workspace / graph / task
```

## 4. 通用字段约定

除 `optimize_result` 和 `operation_log` 外，其余主表建议统一包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| created_by | BIGINT | 创建人 |
| updated_by | BIGINT | 修改人 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 修改时间 |
| deleted | TINYINT(1) | 逻辑删除，0=未删，1=已删 |

说明：

- MyBatis-Plus 可统一接管 `id`、逻辑删除、自动填充时间与审计字段。
- `optimize_result` 是任务产物，可不单独维护 `updated_by`。
- `operation_log` 是审计流水，默认不做逻辑删除。

## 6. Redis 使用边界

Redis 第一版不建表，但建议文档上明确缓存职责：

- 登录会话：Sa-Token 默认会话与 token 存储。
- 流程图详情缓存：
  - `graph:detail:{graphId}`
- 任务状态缓存：
  - `optimize:task:status:{taskId}`
- 任务结果摘要缓存：
  - `optimize:result:summary:{taskId}`

缓存原则：

- 查询优先读缓存；
- 写操作后删除或刷新缓存；
- MySQL 永远是最终事实来源。

## 7. 第一版不做的内容

- 不建 `flow_version`
- 不建复杂 `graph_diff`
- 不建工作空间成员表
- 不建消息队列表
- 不建算法中间步骤结果表

## 8. 建议的建模备注

- `node_code`、`condition_code` 等字符串字段保留，是为了平滑承接旧 YAML 中的 `nodeID / conditionID`。
- 优化算法执行前，需要把数据库中的 `flow_graph + process_node + process_path + constraint_condition + equipment` 转换为算法模型 `ProcessMap`。
- 优化算法执行后，再把结果转换成 `optimize_result.result_graph_json + diff_json + map_code`。

这也是后续 `converter` 层最核心的职责边界。
