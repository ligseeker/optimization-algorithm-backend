# 08_缓存与日志设计说明

## 1. 编辑态与展示态

### 编辑态
编辑态指流程图结构发生变更的操作：
- 节点新增/修改/删除
- 路径新增/删除
- 装备新增/修改/删除
- 约束新增/修改/删除
- YAML 导入入库

编辑态原则：
- 只以 MySQL 为准；
- 不对 Redis 中完整图结构做局部更新；
- 在同一事务中执行 `graph_version + 1`。

### 展示态
展示态指只读查看流程图详情：
- `GET /api/graphs/{graphId}/detail`

展示态原则：
- 先查 `flow_graph.graph_version`；
- 按版本拼接缓存 key；
- 命中返回 Redis，未命中回源 MySQL 聚合后回填 Redis。

## 2. graph_version 设计原因

流程图是聚合对象，节点删除会联动影响路径/约束/指标。  
如果用“局部改缓存”易出现脏结构，因此改为版本化缓存：

`graph:detail:{graphId}:v{graphVersion}`

优点：
- 无需删除旧 key；
- 不需要维护复杂局部更新逻辑；
- 结构一致性更高。

## 3. 缓存策略

### 3.1 图详情缓存
- key：`graph:detail:{graphId}:v{graphVersion}`
- TTL：30 分钟
- 数据来源：MySQL 聚合查询

### 3.2 任务状态缓存
- key：`optimize:task:status:{taskId}`
- 状态：`PENDING/RUNNING/SUCCESS/FAILED`
- 策略：状态变更主动更新；查询未命中回源并回填

### 3.3 优化结果缓存
- key：`optimize:result:{taskId}`
- 策略：先落 MySQL，再写 Redis；同 taskId 结果不可变
- retry 复用 taskId 时：先清理旧结果与旧状态缓存

## 4. Redis 异常降级

- 读 Redis 异常：降级查询 MySQL；
- 写 Redis 异常：记录 WARN，不中断主业务；
- MySQL 始终是最终可信数据源。

## 5. 操作日志设计

### 5.1 注解
- `@OperationLog(operationType, objectType, objectIdParam)`

### 5.2 AOP 记录字段
- `userId`
- `operationType`
- `objectType`
- `objectId`
- `requestMethod`
- `requestUri`
- `costTimeMs`
- `responseCode`
- `successFlag`
- `errorMessage`

### 5.3 安全要求
- 不记录密码、token；
- 失败操作也写日志；
- 日志写入 `operation_log` 表。

## 6. 覆盖范围

关键接口已覆盖：
- 登录
- 创建工作空间
- 创建流程图
- 导入 YAML
- 节点增改删
- 路径新增/删除
- 装备增改删
- 约束增改删
- 提交优化任务
- 查询优化结果
- 导出 YAML
