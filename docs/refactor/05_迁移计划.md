# 迁移计划

## 1. 迁移原则

1. **新增不删旧** — 新接口使用 `/api` 前缀，旧接口保留不动
2. **算法不碰** — `algorithm` 包尽量少改，避免破坏已有算法逻辑
3. **逐步替换** — 5 个阶段迭代推进，每个阶段可独立验证
4. **先基础设施后业务** — 数据库连接、统一返回值、异常处理先行
5. **先存储后任务** — 先完成 CRUD 数据迁移，再做异步任务中心
6. **随时可回滚** — 旧接口和文件存储方式保留，新方案出问题不影响旧功能

public void deleteNode(Long graphId, Long nodeId) {

### Phase 3：YAML 导入导出

**目标**：旧文件数据接入新数据库，YAML 从主存储变为导入/导出格式。

**任务列表**：

- [ ] importfile 模块 Controller、Service
- [ ] GraphConverter（Entity ↔ ProcessMap 双向转换）
- [ ] 导入解析（复用 Main.readData 逻辑）
- [ ] 导入校验（节点唯一、路径存在、约束合法）
- [ ] 导入写入 MySQL
- [ ] 导出（MySQL → YAML 文件下载）
- [ ] 优化结果导出 YAML

**导入关键路径**：

```text
上传 YAML → Main.readData() → InputInfo → ProcessMap
    → 校验（节点ID重复？路径节点存在？约束类型合法？装备名唯一？）
    → GraphConverter.toEntityList(ProcessMap, graphId)
    → 批量写入 MySQL
    → 缓存流程图详情
    → 返回 graphId
```

**与旧接口的关系**：
- 旧 `POST /upload`、`POST /initialize` 仍然可用（通过文件操作）
- 新 `POST /api/import/graphs` 直接将数据写入 MySQL
- 两者数据源不同，互不影响

**验证**：
- 上传一个旧 YAML 文件 → 导入成功 → 返回 graphId
- 根据 graphId 查询详情 → 与 YAML 内容一致
- 根据 graphId 导出 YAML → 与导入内容一致
- 导入不合法的 YAML → 返回具体错误报告

### Phase 4：优化任务中心

**目标**：将原同步算法接口改造成异步任务中心。

**任务列表**：

- [ ] optimize_task Entity、Mapper
- [ ] optimize_result Entity、Mapper
- [ ] 任务提交接口 `POST /api/optimize/tasks`
- [ ] 任务状态查询 `GET /api/optimize/tasks/{taskId}`
- [ ] 任务列表查询 `GET /api/optimize/tasks`
- [ ] 任务重试 `POST /api/optimize/tasks/{taskId}/retry`
- [ ] ThreadPoolTaskExecutor 配置
- [ ] AlgorithmExecutor（Runnable 封装）
- [ ] OptimizeConverter（Entity → ProcessMap）
- [ ] 简化版 Diff 计算（DiffService）
- [ ] 结果查询 `GET /api/optimize/tasks/{taskId}/result`
- [ ] 补偿机制：服务重启后 PENDING 任务的处理策略

**线程池配置**：

```yaml
thread-pool:
  core-size: 2
  max-size: 4
  queue-capacity: 10
  thread-name-prefix: "optimize-"
```

**任务状态管理**：

```text
提交任务 → DB写入 PENDING → Redis写入 PENDING → 提交线程池
    ↓ 线程开始执行
DB改为 RUNNING → Redis改为 RUNNING → 执行算法
    ↓ 执行成功
DB改为 SUCCESS → 写入结果 → Redis缓存结果
    ↓ 执行失败
DB改为 FAILED → 记录 error_message → Redis缓存错误状态
```

**关键安全措施**：
- 不允许在单例 Service 中保存任务上下文（`mapList`、`inputInfo`、`oldPaths`）
- 任务上下文定义为 `OptimizeContext` 局部对象
- 算法执行用 `try-catch` 包裹，异常不扩散
- 线程池拒绝策略：`CallerRunsPolicy` 或记录日志

**验证**：
- 提交优化任务 → 返回 taskId → 轮询状态 → 状态变为 SUCCESS → 查询结果
- 参数错误的任务 → 状态变为 FAILED → 查看错误信息
- 重试失败任务 → 重新执行成功
- 同时提交多个任务 → 各自独立执行，结果正确

- [ ] @OperLog 注解定义

- [ ] operation_log 写入和查询
- [ ] 接口文档完善（Swagger 注解）
- [ ] 补充单元测试

**缓存一致性策略**：

```text
读：Redis → 未命中 → MySQL → 写入 Redis → 返回
写：先写 MySQL → 事务成功后删除 Redis
```

不需要消息队列补偿，第一版只记录日志。

**操作日志实现**：

```java
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface OperLog {
    String operationType();  // CREATE / UPDATE / DELETE / OPTIMIZE / IMPORT / EXPORT
    String objectType();     // GRAPH / NODE / PATH / EQUIPMENT / CONSTRAINT / TASK
}
```

```java
@Aspect
@Component
public class OperLogAspect {
    @Around("@annotation(operLog)")
    public Object around(ProceedingJoinPoint point, OperLog operLog) {
        // 记录请求开始时间
        // 执行原方法
        // 记录用户ID、URI、参数(脱敏)、耗时、结果
        // 异步写入 operation_log
    }
}
```

**验证**：
- 首次查询流程图详情 → 慢查询 → 第二次查询 → 缓存命中
- 新增节点 → 缓存删除 → 再次查询 → 更新后的数据
- 提交优化任务 → 查询任务状态 → 缓存命中
- 操作日志能正常写入和查询

## 4. 数据迁移策略

### 4.1 存量 YAML 数据导入

已有 YAML 文件在 `src/main/resources/tempDir/input/` 和 `temp/` 目录中。迁移方案：

1. 提供迁移工具类 `DataMigrationRunner`（实现 `CommandLineRunner`）
2. 扫描指定目录下的 YAML 文件
3. 解析为 InputInfo → ProcessMap
4. 校验后写入 MySQL
5. 生成 graphId → workspaceId 映射记录

或者通过新导入接口手动导入：

```text
管理员登录 → POST /api/import/graphs → 上传 YAML → 写入数据库
```

### 4.2 数据一致性校验

导入后核对：
- 节点数量一致
- 路径数量一致
- 约束数量一致
- 装备数量一致
- 节点的 time/precision/cost 数值一致

## 5. 回滚方案

| 场景 | 回滚操作 |
|------|----------|
| 新接口出问题 | 前端切回旧接口，新接口修复后再切回来 |
| 数据库出问题 | MySQL dump 恢复，依赖旧文件系统继续运行 |
| 算法执行异常 | 任务状态设为 FAILED，不影响原同步算法接口 |
| 缓存出问题 | 关闭 Redis 缓存，降级为直接查 MySQL |
