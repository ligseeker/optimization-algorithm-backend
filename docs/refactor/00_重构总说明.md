# 00_重构总说明

## 1. 文档目的

本文档用于说明 `optimization-algorithm-backend` 的本轮重构方向、边界与禁止事项，作为后续数据库设计、接口设计、模块拆分与迁移计划的总入口。

## 2. 当前项目现状

当前仓库仍以以下模式为主：

- 流程图与相关数据大量通过 YAML/JSON 文件读写管理；
- 优化逻辑集中在 `algorithm` 包；
- 接口层与服务层已存在，但整体仍偏向文件驱动；
- `src/main/resources/tempDir` 下保留了历史输入输出样例；
- 仓库内已经存在 `docs/refactor/01-06` 系列专题文档，缺少统一的总说明入口。

因此，本轮工作不是推倒重写，而是在保留现有算法资产前提下做平台化、结构化重构。

## 3. 本轮重构目标

本轮重构目标明确为以下五项能力落地：

- `MySQL`：作为流程图、节点、路径、装备、约束、任务、结果等结构化数据的最终数据源；
- `Redis`：作为热点数据、任务状态、短期查询结果等缓存层；
- `MyBatis-Plus`：作为数据访问层基础能力；
- `Sa-Token`：作为登录认证与权限校验基础设施；
- `ThreadPoolTaskExecutor`：作为优化任务异步执行基础设施。

## 4. 重构原则

- 保持增量重构，避免一次性整体替换现有项目；
- 尽量少改 `algorithm` 包中的核心算法；
- 通过新增 `entity`、`dto`、`vo`、`mapper`、`converter`、`service` 等层次完成平台化改造；
- 旧接口可暂时保留，新接口统一走 `/api` 前缀；
- 新接口统一返回 `Result<T>`；
- 所有写操作要纳入事务控制；
- 所有资源访问都要绑定当前用户权限校验；
- Redis 只做缓存，MySQL 才是最终事实来源。

## 5. 明确禁止事项

以下事项在后续重构中明确禁止：

- 不要直接删除 `algorithm` 包；
- 不要把算法模型直接改成 Entity；
- 不要让 Controller 直接调用算法；
- 不要在单例 Service 中保存任务上下文；
- YAML 只作为导入导出格式，不能继续作为主存储格式。

## 6. 后续实施建议

建议后续按以下顺序推进：

1. 先完成领域建模与数据库表设计。
2. 再补齐 Entity、DTO、VO、Mapper、Converter 分层。
3. 然后接入 Sa-Token、Redis 与统一结果结构。
4. 最后将优化任务改造成基于线程池的异步任务中心，并补齐导入导出、结果查询与操作日志。
