# 测试计划

## 1. 测试目标

- 确保重构后的数据库操作与原文件操作结果一致
- 确保算法在异步任务中的执行结果与原同步执行一致
- 确保 CRUD 操作的事务正确性
- 确保权限校验不遗漏
- 确保缓存与数据库一致性
- 确保核心流程端到端可运行

---

## 2. 测试策略

### 2.1 分层测试

| 层次 | 工具 | 范围 | 目标 |
|------|------|------|------|
| 单元测试 | JUnit + Mockito | Service、Converter、Validator | 业务逻辑正确性 |
| 集成测试 | SpringBootTest | Mapper、Repository | 数据库读写正确性 |
| 接口测试 | Apifox / Postman / MockMvc | Controller | 接口请求响应正确性 |
| 端到端测试 | 手动 + 脚本 | 完整业务流程 | 核心链路可走通 |

### 2.2 测试数据

- 使用原项目中的 `input30.yaml`、`input208.yaml`、`input234.yaml` 等文件作为测试输入
- 算法执行结果与原 `temp/output/` 目录中的结果进行比对
- 数据库测试使用 H2 内存数据库（可选）或独立的 test MySQL

---

## 3. 单元测试

### 3.1 重点测试模块

| 模块 | 测试内容 |
|------|----------|
| GraphConverter | Entity ↔ ProcessMap 字段映射正确性 |
| ImportValidator | 节点重复、路径节点不存在、约束类型非法等校验 |
| DiffService | 路径新增/删除对比、指标变化计算 |
| OptimizeTaskService | 任务状态机流转、并发任务隔离 |
| CacheService | 缓存读写、失效、更新 |
| OperLogAspect | 日志记录正确性 |

### 3.2 测试用例

#### GraphConverter

```text
测试 1：ProcessMap → Entity 转换
  - 输入：包含 3 个节点、2 条路径、1 个约束、1 个装备的 ProcessMap
  - 验证：生成的 Entity 列表字段值正确，graph_id 正确

测试 2：Entity → ProcessMap 转换
  - 输入：从 DB 查询的 Entity 列表
  - 验证：生成的 ProcessMap.ProcessNodes 数量正确，字段值正确

测试 3：空图转换
  - 输入：无节点、无路径、无约束、无装备的 ProcessMap
  - 验证：不报错，返回空的 Entity 列表
```

#### ImportValidator

```text
测试 1：节点 ID 重复 → 返回错误，列出重复节点
测试 2：路径起点不存在 → 返回错误，列出缺失节点
测试 3：路径终点不存在 → 返回错误
测试 4：约束关联节点不存在 → 返回错误
测试 5：约束类型非法 → 返回错误
测试 6：合法的 YAML → 校验通过
测试 7：空的 YAML 文件 → 返回错误
```

#### DiffService

```text
测试 1：新增一条路径 → addedPaths 包含该路径
测试 2：删除一条路径 → removedPaths 包含该路径
测试 3：路径不变 → addedPaths 和 removedPaths 均为空
测试 4：时间降低 → metricDiff.time.change 为负数
测试 5：精度提升 → metricDiff.precision.change 为正数
测试 6：成本不变 → metricDiff.cost.change 为 0
```

#### OptimizeTaskService

```text
测试 1：提交任务 → 状态 PENDING
测试 2：任务开始执行 → 状态 RUNNING
测试 3：任务执行成功 → 状态 SUCCESS，有结果数据
测试 4：任务执行失败 → 状态 FAILED，有 error_message
测试 5：重试失败任务 → 重新变为 PENDING
测试 6：并发提交多个任务 → 任务 ID 唯一，context 隔离
```

#### CacheService

```text
测试 1：查流程图详情 → 未命中 → 查 MySQL → 写入缓存 → 返回
测试 2：再查流程图详情 → 缓存命中 → 直接返回
测试 3：更新节点 → 删除缓存 → 再查 → 重新写入
测试 4：缓存 Key 格式正确（graph:detail:{id}）
```

### 5.2 核心业务流

```text

2. 创建工作空间
3. 创建流程图
4. 新增 3 个节点

6. 新增 1 个约束
7. 新增 1 个装备

9. 删除一个节点 → 确认关联路径/约束已删除

```

### 5.3 优化链路测试

```text
完整优化链路：
1. 导入 YAML（POST /api/import/graphs）
2. 提交优化任务（POST /api/optimize/tasks）
3. 轮询任务状态直到 SUCCESS（GET /api/optimize/tasks/{taskId}）
4. 查询优化结果（GET /api/optimize/tasks/{taskId}/result）
5. 验证 Diff 数据不为空
6. 导出优化结果 YAML（GET /api/export/tasks/{taskId}/yaml）
```

## 6. 数据一致性验证

### 6.1 新旧接口结果比对

导入 YAML 后，用新接口查 MySQL 的数据，与旧接口读文件的数据对比：

```text
nodes: 数量一致，每个 node_code、time、precision、cost 一致
paths: 数量一致，每条 start-end 关系一致
constraints: 数量一致，每个 conditionType、node 关系一致
equipments: 数量一致，每个 name、color 一致
```

### 6.2 算法结果比对

导入相同的 YAML 数据，比较原同步算法和异步算法的输出：

```text
graph 结构：节点、路径、指标一致
Diff 结果：新增/删除路径一致
mapCode：Mermaid 代码一致
```

## 7. 性能测试

### 7.1 目标

| 场景 | 目标 |
|------|------|
| 流程图详情查询（缓存命中） | < 100ms |
| 任务状态查询（缓存命中） | < 50ms |
| 普通分页查询 | < 300ms |
| YAML 导入（中等规模） | 稳定完成 |

### 7.2 关注点

```text
- 流程图详情首次查询（未缓存）的响应时间
- 同时提交多个优化任务时线程池的表现
- 大量节点（200+）的流程图查询性能
```
