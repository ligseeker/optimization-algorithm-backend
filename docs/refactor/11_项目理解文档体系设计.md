# 11_项目理解文档体系设计

## 1. 背景与目标

当前仓库已经完成从“基于 YAML/JSON 文件的流程图与优化算法项目”向“基于 Spring Boot + MyBatis-Plus + MySQL + Redis + Sa-Token 的流程优化任务管理平台”的渐进式重构。

现有 `docs/refactor/` 更偏向“重构设计、迁移方案、接口与数据结构说明”，但缺少一套专门面向以下场景的理解型文档体系：

- 帮助后续自己快速重新进入项目上下文；
- 帮助自己按业务模块彻底看懂实现逻辑；
- 帮助从源码理解中提炼出简历表达；
- 帮助围绕架构、缓存、权限、异步任务、算法接入等技术点准备面试问答。

因此，本次新增一套独立的 `docs/project-understanding/` 文档体系，用于承载“项目理解、技术复盘、简历提炼、面试准备”四类信息。

## 2. 设计原则

### 2.1 面向未来复盘的自己

文档默认读者不是陌生团队成员，而是“未来重新看这个项目的自己”。因此文档重点应放在：

- 为什么这么设计；
- 一次请求在系统里如何流转；
- 模块边界在哪里；
- 容易混淆和容易遗忘的点有哪些；
- 哪些内容适合对外表达，哪些只适合内部理解。

### 2.2 按业务模块组织，不按技术分层平铺

主文档层按业务模块拆分，而不是按 Controller / Service / Mapper / Converter 平铺，原因如下：

- 更符合后续复盘时的认知路径；
- 更方便围绕具体业务链路理解代码；
- 更利于提炼简历项目亮点与面试答题素材。

技术分层相关内容统一收敛到专题文档中，避免重复。

### 2.3 一套底座，多场景复用

整套文档采用“三层结构”：

- 总览层：帮助快速重建全局认知；
- 模块层：作为核心理解底座，按业务模块讲透；
- 专题层：把缓存、权限、事务、异步任务、算法接入等跨模块技术点集中讲透；
- 输出层：从前面三层中提炼简历表达与面试问答。

这样可以避免“为了简历写一套、为了面试写一套、为了复盘再写一套”的重复劳动。

## 4. 模块划分依据

主业务模块采用当前仓库 `src/main/java/com/example/optimization_algorithm_backend/module/` 下的真实边界：

- `auth`
- `workspace`
- `graph`
- `node`
- `path`
- `equipment`
- `constraint`
- `yaml`
- `optimize`
- `log`

以下内容不单独作为业务模块文档，而作为专题内容解释：

- `config`
- `common`
- `infrastructure/persistence`
- `algorithm`

原因是这些目录更偏向横向基础设施、通用能力和适配层，而不是用户视角下的独立业务模块。

## 5. 业务模块文档模板

每篇业务模块文档统一采用以下结构：

1. 模块定位
2. 核心对象与数据边界
3. 对外接口
4. 核心调用链
5. 关键设计动机
6. 技术点拆解
7. 权限、事务、缓存、异常
8. 与其他模块的关系
9. 容易混淆或踩坑的点
10. 简历表达与面试追问

统一模板的目标是：

- 让不同模块之间具备可横向比较的结构；
- 让后续复盘时能快速定位到想找的信息；
- 让“源码理解”和“对外表达提炼”天然连通。

## 6. 文档职责分工

### 6.1 总览层

作用是重建全局认知，不追求细节穷尽。

- `00_阅读指南与文档地图.md`
  - 说明整套文档怎么读；
  - 给出不同使用场景下的推荐阅读路径。
- `01_项目全景与重构主线.md`
  - 说明项目从哪里来、为什么重构、重构后变成什么。
- `02_整体架构与核心调用链.md`
  - 说明分层架构、主链路、模块关系。
- `03_横向技术专题总览.md`
  - 作为专题入口索引。

### 6.2 模块层

作用是按业务模块建立“真正可复盘”的底座文档。

### 6.3 专题层

作用是把跨模块技术点集中解释，避免在多个业务模块文档中大面积重复。

### 6.4 输出层

作用是从前面三层中提炼：

- 简历项目表达；
- 面试高频问答；
- 可被追问点与回答边界。

## 8. 本轮落地范围

本轮只做文档准备工作，不改 Java 业务代码。

本轮交付内容：

- 在 `docs/refactor/` 下补一份文档体系设计说明；
- 在 `docs/project-understanding/` 下创建新文档目录；
- 创建总览层、模块层、专题层全部目标文件；
- 先为每个文件补齐职责说明、写作提纲和后续填充方向。

## 9. 后续维护约定

后续补写正文时应遵循以下规则：

- 尽量结合真实代码路径、类名、接口和数据结构；
- 优先写“调用链”和“设计动机”，避免只做静态目录介绍；
- 对于缓存、事务、权限、线程池、算法接入等点，先在模块里写“本模块如何落地”，再在专题里做横向总结；
- 简历表达必须以真实实现为依据，避免写出经不起追问的表述。

## 10. 自检结论

本设计具备以下特征：

- 范围聚焦：只新增理解型文档，不触碰业务代码；
- 与现有重构文档互补，不替代 `docs/refactor/`；
- 目录清晰，可持续补写；
- 同时服务于复盘、简历和面试三个场景。
