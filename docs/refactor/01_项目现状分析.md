# 01_项目现状分析

## 1. 分析范围

本文档基于当前仓库源码现状整理，重点覆盖以下内容：

- Controller 层接口清单；
- Service 层核心实现与职责；
- 当前 YAML / JSON 文件读写链路；
- `Algorithm1 / Algorithm2 / Algorithm3` 的调用方式；
- `ProcessMap / MultiNode / ProcessPath / Equipment / ConstraintCondition` 的职责；
- 当前代码中的并发风险、职责混杂点、扩展性问题；
- 迁移到 `MySQL + Redis` 时的优先改造模块；
- “可以保留的代码”与“必须重构的代码”清单。

本轮仅做现状分析，不涉及 Java 代码修改。

## 2. 当前项目总体结构

当前后端仍然是“文件驱动 + 算法调用”的结构，核心代码集中在以下目录：

- `src/main/java/com/example/optimization_algorithm_backend/Controller`
  - 提供流程图编辑、优化、目录、导入导出、装备、约束等 HTTP 接口。
- `src/main/java/com/example/optimization_algorithm_backend/Service`
  - 提供与 Controller 对应的服务接口。
- `src/main/java/com/example/optimization_algorithm_backend/Service/Impl`
  - 主要负责 YAML 文件读写、流程图对象转换、算法调度、ZIP 导入导出。
- `src/main/java/com/example/optimization_algorithm_backend/algorithm`
  - 保存 `Algorithm1/2/3`、`Main` 工具入口、算法模型、样例 YAML/JSON/CSV。
- `src/main/resources/application.yml`
  - 配置了 `history/input/output/temp/equipment` 等目录路径。
- `src/main/resources/tempDir`
  - 保留了一批历史输入输出样例。

从代码职责上看，项目还没有形成数据库驱动的分层结构，数据生命周期主要围绕以下目录展开：

- `temp/history/`：正在编辑或上传后的原始流程图；
- `temp/input/`：保存后的待优化实例；
- `temp/output/`：优化结果；
- `temp/temp/`：算法执行中间流程图；
- `temp/images/equipment/`：装备图片。

## 3. Controller 层接口现状

### 3.1 `AlgorithmController`

文件：`src/main/java/com/example/optimization_algorithm_backend/Controller/AlgorithmController.java`

当前接口：

- `POST /optimizeByFile`
  - 上传 YAML/JSON 文件，转为 `ProcessMap` 后直接执行 `Algorithm1`。
- `POST /optimizeByInput`
  - 接收 `Map<String, Object>` 输入，理论上转为 `ProcessMap` 后执行 `Algorithm1`。
  - 但当前 `AlgorithmServiceImpl.inputToMap()` 还是空实现，实际上不可用。
- `POST /uploadFile`
  - 上传文件后解析流程图并返回 `mapCode` 与旧值。
- `GET /downloadFile`
  - 按 `history/input/output/temp` 类型下载 YAML 文件。
- `GET /hello`
  - 测试接口。

现状特征：

- Controller 直接 `new Algorithm1()`；
- 返回值统一为 `Map<String, Object>`；
- 下载逻辑直接暴露底层目录结构；
- `optimizeByInput` 对应的核心转换尚未实现完整。

### 3.2 `MapController`

文件：`src/main/java/com/example/optimization_algorithm_backend/Controller/MapController.java`

当前接口：

- `POST /getRandomMap`
  - 读取模板 YAML，随机生成约束和数值，再写入 `history`。
- `POST /upload`
  - 上传 YAML 文件进入 `history`。
- `POST /initialize`
  - 初始化空白 `ProcessMap` 并写入 `history`。
- `POST /save`
  - 将 `history` 中的流程图另存到 `input`。
- `POST /optimize`
  - 从 `input` 读取流程图，检查约束后，根据 `algorithmType1/algorithmType2` 选择 `Algorithm1/2/3` 执行优化，并输出到 `output`。
- `GET /download`
  - 下载 `output` 下的 YAML 文件。
- `GET /getTemplate`
  - 下载模板 YAML。

这是当前最核心的流程图与优化入口控制器，负责：

- 文件路径拼装；
- 输入参数解析；
- 约束合法性校验；
- 算法类型路由；
- 优化收益计算；
- 输出 `mapCode` 与 `values`。

也因此，它已经承担了过多编排职责。

### 3.3 `NodeController`

- `POST /addNode`
- `POST /updateNode`
- `POST /deleteNode`
- `GET /getNodeInfo`

- `GET /getUnassignedNodes`

主要负责节点的增删改查，以及节点与装备之间的关联维护。

- 节点与装备关系维护部分在 Controller 里手工完成；
- Controller 内含明显业务逻辑，而不是单纯参数接收。

### 3.4 `PathController`

文件：`src/main/java/com/example/optimization_algorithm_backend/Controller/PathController.java`

- `POST /addPath`
- `POST /deletePath`

### 3.5 `ConstraintController`

### 3.6 `EquipmentController`

文件：`src/main/java/com/example/optimization_algorithm_backend/Controller/EquipmentController.java`

## 4. Service 层核心实现现状

### 4.1 `AlgorithmServiceImpl`

文件：`src/main/java/com/example/optimization_algorithm_backend/Service/Impl/AlgorithmServiceImpl.java`

当前承担的职责：

- `saveFile`
  - 保存上传的 `.yaml` 文件；
  - 对 `.json` 文件先调用 `Main.readData1()` 转为 `InputInfo`，再转成 YAML。
- `pathToMap`
  - 调用 `Main.readData()` 读取 YAML，转成 `InputInfo`，再调用 `Main.initMapTest()` 转为 `ProcessMap`。
- `optimizeMap1 / optimizeMap2 / optimizeMap3`
  - 组织算法执行流程：
    1. 创建 `temp/` 中间文件；
    2. 记录原始路径；
    3. 调用 `utils.initProcessMap()` 做约束合并；
    4. 调用算法类的 `getOptimizationMap()`；
    5. 调用 `utils.restoreProcessMap()` 恢复原流程结构；
    6. 调用算法类的 `OptimizeMap()`；
    7. 输出最终 YAML 到 `output/`。
- `checkMap`
  - 按约束类型检查流程图是否满足规则。
- `mapToOutput`
  - 将 `ProcessMap` 转换成前端需要的 `Map<String, Object>` 响应结构。
- `getMapCode`
  - 生成 Mermaid `flowchart LR` 文本。

这是当前最关键的服务实现，也是未来“优化任务中心”的前身。

### 4.2 `NodeServiceImpl`

文件：`src/main/java/com/example/optimization_algorithm_backend/Service/Impl/NodeServiceImpl.java`

职责：

- 在 `history` YAML 中增加、删除、修改节点；
- 读取不同目录中的节点；
- 查询节点关联约束。

实现特点：

- 基于 `AlgorithmService.pathToMap()` 把 YAML 读成 `ProcessMap` 后再修改；
- 节点更新是“先删后加”，没有保留原有节点顺序；
- `updateNodeInMap` 会把 `equipmentName` 直接重置为 `""`，存在数据丢失风险。

### 4.3 `PathServiceImpl`

文件：`src/main/java/com/example/optimization_algorithm_backend/Service/Impl/PathServiceImpl.java`

职责：

- 增删路径；
- 查询路径是否存在；
- 返回某节点的下一层节点列表。

实现特点：

- 直接用 SnakeYAML 把 YAML 读成 `Map<String, Object>`；
- 直接修改 `Paths` 数组后落盘；
- 没有统一复用 `ProcessMap` 模型。

### 4.4 `ConstraintServiceImpl`

文件：`src/main/java/com/example/optimization_algorithm_backend/Service/Impl/ConstraintServiceImpl.java`

职责：

- 约束增删改查；
- 根据节点 ID 判断节点是否存在。

实现特点：

- 同样直接以 `Map<String, Object>` 形式操作 YAML；
- 约束实体是手工拼接 `conditionID/conditionType/description`；
- 对 YAML 中字段顺序做了一点保存格式处理，但没有真正形成领域模型层。

### 4.5 `EquipmentServiceImpl`

文件：`src/main/java/com/example/optimization_algorithm_backend/Service/Impl/EquipmentServiceImpl.java`

职责：

- 装备增删改查；
- 装备重命名时同步修改 `ProcessNodes.equipmentName`；
- 上传装备图片时同步更新 `imagePath`。

实现特点：

- 仍是直接用 `Map<String, Object>` 读写 YAML；
- 同时管理装备和节点之间的关联字段；
- 同时处理元数据与图片文件引用。

### 4.6 `ExampleServiceImpl`

文件：`src/main/java/com/example/optimization_algorithm_backend/Service/Impl/ExampleServiceImpl.java`

职责：

- ZIP 导入；
- ZIP 导出；
- 从输出文件名解析优化配置；
- 生成临时 `configuration.yaml`；
- 校验 ZIP 中 `configuration.yaml`。

实现特点：

- 通过 ZIP 内固定目录结构和命名约定推断业务元数据；
- 当前更像“文件导入导出工具服务”，还不是领域服务。

## 5. 当前 YAML / JSON 文件读写方式

### 5.1 配置入口

文件：`src/main/resources/application.yml`

当前配置路径：

- `path.historyPath: temp/history/`
- `path.inputPath: temp/input/`
- `path.outputPath: temp/output/`
- `path.equipmentPath: temp/images/equipment/`
- `path.tempPath: temp/temp/`
- `upload_dir.temp: temp/`

说明当前系统的主存储仍然是本地文件系统，而不是数据库。

### 5.2 YAML 结构

参考文件：

- `src/main/resources/templateFile.yaml`
- `src/main/resources/tempDir/input/input30-test.yaml`

核心结构包括：

- `ProcessNodes`
- `ConstraintConditions`
- `Paths`
- `Equipments`

其中：

- `ProcessNodes` 保存节点属性；
- `ConstraintConditions` 保存约束；
- `Paths` 保存边；
- `Equipments` 保存装备及其包含的节点列表。

现状观察：

- YAML 中既保存流程结构，也保存装备与节点关联；
- 样例文件中已经出现重复 `nodeID` 的情况，例如 `input30-test.yaml` 中 `Ib` 出现两次，说明当前存储层没有唯一性约束；
- 这会在迁移到数据库时直接转化为数据清洗和约束落地问题。

### 5.3 JSON 读入路径

JSON 只在上传流程中被兼容支持：

- `AlgorithmServiceImpl.saveFile()`
  - 如果扩展名是 `.json`，读取文件内容；
  - 调用 `Main.readData1(String jsonStr)` 解析为 `InputInfo`；
  - 再通过 `Main.initMapTest()` 转为 `ProcessMap`；
  - 最后落成 YAML 文件。

也就是说：

- JSON 不是系统内部长期使用的格式；
- JSON 只是一个临时导入入口；
- 实际内部持久化仍然落到 YAML。

### 5.4 YAML 读写的三种实现方式

当前仓库里存在三套并行的文件读写方式：

1. `Main.readData / Main.writeData`
   - 把 YAML 转为 `InputInfo / ProcessMap` 等强类型对象；
   - 主要用于算法入口和 `AlgorithmServiceImpl`。
2. `PathServiceImpl / ConstraintServiceImpl / EquipmentServiceImpl`
   - 用 SnakeYAML 直接把 YAML 转为 `Map<String, Object>` 后原位修改；
   - 主要用于 CRUD 接口。
3. `ExampleServiceImpl`
   - 直接按 ZIP 和 YAML 文件路径操作文件，不经过统一领域对象。

这说明当前项目没有统一的“仓储层/转换层”。

## 6. `Algorithm1 / Algorithm2 / Algorithm3` 的调用方式

### 6.1 当前调用链

主要入口在 `MapController.optimize()`：

1. 从 `inputPath + example + ".yaml"` 读取输入文件；
2. `algorithmService.pathToMap(path)` 转为 `ProcessMap`；
3. `algorithmService.checkMap(oldMap)` 检查约束；
4. 根据 `algorithmType1` 决定调用：
   - `Algorithm1`
   - `Algorithm2`
   - `Algorithm3`
5. 每个算法实例都先执行 `initAlgorithm(oldMap)`；
6. 再进入 `AlgorithmServiceImpl.optimizeMap1/2/3()` 做统一编排；
7. 算法执行结果写入 `output/*.yaml`，中间过程写入 `temp/*.yaml`；
8. 最后由 Controller 计算 `ratio`、`values`、`mapCode` 并返回。

### 6.2 三个算法类的共性

文件：

- `algorithm/algorithm1/Algorithm1.java`
- `algorithm/algorithm2/Algorithm2.java`
- `algorithm/algorithm3/Algorithm3.java`

公共方法模式基本一致：

- `initAlgorithm(ProcessMap map)`
  - 生成路径邻接表；
  - 计算拓扑序；
  - 记录原始 `time/precision/cost`。
- `getOptimizationMap(...)`
  - 优化链路结构，主要围绕 `FOLLOW`、拓扑关系和路径重构展开。
- `OptimizeMap(...)`
  - 在 `SAME` 约束下尝试节点替换，计算收益。
- `getValue(ProcessMap map)`
  - 返回优化前后的评价值。

### 6.3 `AlgorithmServiceImpl` 中的统一编排流程

无论调用哪个算法，服务层流程都基本一致：

1. 读取原始 `ProcessMap`；
2. 保存 `oldPaths`；
3. 调用 `utils.initProcessMap(inputInfo)`：
   - 将 `ProcessNode` 转成 `MultiNode`；
   - 对 `CONNECT` 等约束做节点合并；
4. 调用算法类的 `getOptimizationMap(...)`：
   - 优化链路结构；
5. 调用 `utils.restoreProcessMap(...)`：
   - 把合并节点恢复成原始结构；
6. 调用算法类的 `OptimizeMap(...)`：
   - 做基于 `SAME` 约束的替换优化；
7. 落盘：
   - 中间图写到 `temp/`
   - 最终图写到 `output/`

### 6.4 现状结论

当前算法调用已经具备“算法引擎 + 执行编排”的雏形，但存在两个关键问题：

- 算法选择逻辑还在 Controller 中；
- 执行过程依赖 `AlgorithmServiceImpl` 的单例成员变量，天然不适合并发任务化。

## 7. 核心模型职责分析

### 7.1 `ProcessMap`

文件：`algorithm/model/ProcessMap.java`

职责：

- 表示一个完整流程图聚合；
- 聚合以下核心信息：
  - `multiNodes`
  - `processPaths`
  - `constraintConditions`
  - `equipments`
  - `totalTime / totalPrecision / totalCost`

它是当前算法和文件读写之间最关键的中间对象。

### 7.2 `MultiNode`

文件：`algorithm/model/MultiNode.java`

职责：

- 表示算法运行时使用的节点模型；
- 既可以表示普通节点，也可以表示约束合并后的“复合节点”；
- `nodeList` 用于保存被合并的原节点；
- `constant` 表示该节点是否由 `CONNECT / CALL / NORMAL` 等关系形成；
- 还附带时间、精度、成本、装备名称等属性。

说明：

- `MultiNode` 不是单纯的持久化节点；
- 它是算法过程中的“运行态节点模型”。

### 7.3 `ProcessPath`

文件：`algorithm/model/ProcessPath.java`

职责：

- 表示流程图中的一条有向边；
- 记录 `startNodeID -> endNodeID`；
- 在算法路径调整、拓扑排序、流程恢复中被频繁使用。

### 7.4 `Equipment`

文件：`algorithm/model/Equipment.java`

职责：

- 表示装备元数据；
- 记录：
  - `name`
  - `nodes`
  - `color`
  - `description`
  - `imagePath`

当前它既承担“装备主数据”，也承担“装备与节点的关联关系”。

### 7.5 `ConstraintCondition`

文件：`algorithm/model/ConstraintCondition.java`

职责：

- 表示两个节点之间的约束关系；
- 记录：
  - `conditionID`
  - `conditionDescription`
  - `conditionType`
  - `nodeID1`
  - `nodeID2`

约束类型定义在 `algorithm/model/Constant.java` 中，包括：

- `CONNECT`
- `SAME`
- `CONTAIN`
- `FOLLOW`
- `CALL`
- `PARTICIPATE`
- `NORMAL`

### 7.6 模型层补充说明

还存在两个配套模型：

- `ProcessNode`
  - 更接近输入文件中的原始节点模型；
- `InputInfo`
  - 更接近“文件导入后的原始数据容器”。

当前模型关系基本是：

`YAML/JSON -> InputInfo -> ProcessMap -> Algorithm1/2/3`

这条链路说明算法模型体系本身是可复用的，但不适合作为未来数据库实体直接复用。

## 8. 当前存在的并发风险、职责混杂与扩展问题

### 8.1 并发风险

#### 8.1.1 `AlgorithmServiceImpl` 使用单例可变状态

文件：`Service/Impl/AlgorithmServiceImpl.java`

以下字段属于 Spring 单例 Service 的共享状态：

- `public ArrayList<ProcessMap> mapList`
- `public InputInfo inputInfo`
- `public LinkedList<ProcessPath> oldPaths`

风险：

- 多个请求同时执行优化时，这些字段会互相覆盖；
- `pathToMap()` 会修改共享 `inputInfo`；
- `optimizeMap1/2/3()` 又依赖这个共享 `inputInfo` 去执行 `utils.initProcessMap(inputInfo)`；
- 如果并发请求交叉进入，A 请求可能读到 B 请求的 `inputInfo`。

这是当前最明确、最严重的并发问题之一。

#### 8.1.2 所有 YAML 文件读写都缺少锁和事务

涉及类：

- `NodeServiceImpl`
- `PathServiceImpl`
- `ConstraintServiceImpl`
- `EquipmentServiceImpl`
- `ExampleController`
- `ExampleServiceImpl`

风险：

- 两个请求同时修改同一 `history/*.yaml` 时，后写入者会覆盖先写入者；
- 当前没有文件锁、版本校验、事务补偿；
- 重命名与删除操作可能和优化、下载、编辑操作并发冲突。

#### 8.1.3 `temp/` 中间文件清理逻辑会互相影响

文件：`AlgorithmServiceImpl.optimizeMap1/2/3`

当前逻辑会：

- 列出 `temp/` 目录文件；
- 当文件数达到阈值后，根据最后修改时间排序；
- 条件不满足时直接删除前 5 个文件。

风险：

- 不同请求的中间文件共用同一个目录；
- 一个请求可能删掉另一个请求的中间结果；
- 这对未来异步任务中心和结果查询非常不友好。

#### 8.1.4 算法类大量原地修改输入对象

例如：

- `Algorithm1.updateProcessPaths()`
- `Algorithm2.RandomCreateMap()`
- `Algorithm2.IterativeOptimize()`
- `Algorithm3.RandomCreateMap()`
- `utils.initProcessMap()`
- `utils.restoreProcessMap()`

问题：

- 很多方法直接复用并修改传入的 `ProcessMap`、`LinkedList<ProcessPath>`、`ArrayList<ConstraintCondition>`；
- 方法看起来像“返回新对象”，实际上经常是原地修改；
- 这会放大并发风险，也让调试与回溯变难。

### 8.2 职责混杂问题

#### 8.2.1 Controller 过重

典型类：

- `MapController`
- `NodeController`
- `EquipmentController`
- `DirectoryController`

具体表现：

- 参数解析、业务校验、目录操作、文件命名、算法选择、结果组装都写在 Controller 中；
- `MapController.optimize()` 甚至直接决定实例化哪个算法；
- `DirectoryController` 负责目录扫描、算法名称映射、输出文件名解析、收益计算。

这与后续“Controller 只负责参数接收和响应”的目标不符。

#### 8.2.2 Service 风格不统一

当前存在两种服务实现风格：

1. 面向 `ProcessMap` 的强类型风格
   - `AlgorithmServiceImpl`
   - `NodeServiceImpl`
2. 面向 `Map<String, Object>` 的 YAML 原地修改风格
   - `PathServiceImpl`
   - `ConstraintServiceImpl`
   - `EquipmentServiceImpl`

结果是：

- 同一领域对象有时走模型对象，有时走原始 Map；
- 很难插入统一的 DTO/VO/Converter；
- 未来迁移数据库时，改造成本会更高。

#### 8.2.3 `Main` 变成了杂糅式工具类

文件：`algorithm/Main.java`

同时承担了：

- YAML/JSON 转换；
- `ProcessMap` 初始化；
- YAML 输出；
- 差异比较；
- Mermaid `mapCode` 生成；
- 路径搜索工具。

这会导致：

- 工具能力与业务能力耦合；
- 未来很难清晰拆到 converter、import/export、diff、visualization 等独立模块。

### 8.3 难以扩展的问题

#### 8.3.1 目录名和文件名承担了业务标识职责

例如输出文件命名规则：

`example_time_precision_cost_algorithmType1_algorithmType2.yaml`

问题：

- 业务元数据靠文件名编码；
- 查询、筛选、权限控制、任务追踪都依赖字符串解析；
- 一旦要支持工作空间、用户隔离、任务历史、结果分页，文件命名方案会迅速失控。

#### 8.3.2 没有任务概念

当前“优化”只是一次同步请求：

- 没有任务表；
- 没有任务状态；
- 没有任务日志；
- 没有失败重试；
- 没有运行上下文隔离。

这与后续“任务中心 + 状态查询 + 异步执行”目标存在明显差距。

#### 8.3.3 缺少统一的领域边界

当前“流程图管理”实际拆散在多个控制器和服务里：

- 节点；
- 路径；
- 约束；
- 装备；
- 示例导入导出；
- 输出目录浏览。

但这些其实都属于同一个流程图聚合上下文，当前是以“文件片段修改”的方式分散处理的，未来很难安全扩展。

## 9. 迁移到 MySQL + Redis 时的优先改造模块

### 9.1 第一优先级：流程图持久化与仓储抽象

优先改造对象：

- `AlgorithmServiceImpl.pathToMap`
- `NodeServiceImpl`
- `PathServiceImpl`
- `ConstraintServiceImpl`
- `EquipmentServiceImpl`
- `MapController.save / initialize / upload`

原因：

- 当前所有结构化数据都依赖 YAML；
- 这是迁移到 MySQL 的第一阻塞点；
- 必须先建立新的 `entity + mapper + converter + service` 结构，才能让流程图、节点、路径、装备、约束脱离文件主存储。

建议方向：

- 引入流程图聚合根；
- 将 `ProcessMap` 保留为算法模型；
- 新建数据库实体，如：
  - `workspace`
  - `process_graph`
  - `process_node`
  - `process_path`
  - `equipment`
  - `constraint_condition`
- 再通过 Converter 在 Entity 和算法模型之间转换。

### 9.2 第二优先级：优化执行编排与任务中心

优先改造对象：

- `MapController.optimize`
- `AlgorithmController.optimizeByFile / optimizeByInput`
- `AlgorithmServiceImpl.optimizeMap1/2/3`

原因：

- 当前优化是同步 HTTP 调用；
- `AlgorithmServiceImpl` 有明显单例共享状态问题；
- 后续引入 `ThreadPoolTaskExecutor` 时，必须把“任务创建、任务执行、任务状态、任务结果”拆开。

建议方向：

- 新增 `OptimizationTask` 领域对象和任务表；
- `Controller` 只创建任务，立即返回 `taskId`；
- 线程池异步执行算法；
- Redis 缓存任务状态，MySQL 持久化任务结果与日志；
- 每次任务执行时创建独立上下文对象，禁止复用单例字段。

### 9.3 第三优先级：导入导出适配层

优先改造对象：

- `AlgorithmServiceImpl.saveFile`
- `ExampleServiceImpl`
- `MapController.getTemplate / download`
- `AlgorithmController.downloadFile`

原因：

- YAML / JSON / ZIP 未来只能是导入导出格式；
- 不能继续承担主存储职责；
- 但可以保留为边界适配器。

建议方向：

- 把当前 `Main.readData / readData1 / writeData` 提炼为 `YamlImportExportService`；
- ZIP 逻辑保留为 `ExampleArchiveService`；
- 内部读写改为“数据库 <-> 算法模型 <-> YAML 导出”。

### 9.4 第四优先级：目录与示例浏览逻辑

优先改造对象：

- `DirectoryController`
- `ExampleController`

原因：

- 当前目录扫描是围绕本地文件系统；
- 未来应围绕工作空间、流程图、任务、结果来查询；
- 文件名解析不应再承担业务查询职责。

建议方向：

- 以数据库查询替代 `File.listFiles()`；
- 把“示例目录”重构成：
  - 工作空间列表；
  - 流程图列表；
  - 任务列表；
  - 结果列表。

### 9.5 第五优先级：权限、缓存、日志

当前仓库几乎没有真正的：

- 用户体系；
- 资源权限；
- 操作日志；
- 缓存设计。

这部分需要新增而不是迁移，但要建立在前面几个核心模块稳定之后。

## 10. 可以保留的代码

以下代码建议尽量保留，采用“包装 + 适配 + 转换”的方式复用：

### 10.1 算法核心实现

- `algorithm/algorithm1/Algorithm1`
- `algorithm/algorithm2/Algorithm2`
- `algorithm/algorithm3/Algorithm3`

保留理由：

- 三个算法类已经形成清晰的公共入口；
- 核心计算逻辑独立性相对较高；
- 当前重构边界明确要求“algorithm 包中的核心算法尽量少改”。

### 10.2 算法模型

- `algorithm/model/ProcessMap`
- `algorithm/model/MultiNode`
- `algorithm/model/ProcessPath`
- `algorithm/model/Equipment`
- `algorithm/model/ConstraintCondition`
- `algorithm/model/ProcessNode`
- `algorithm/model/InputInfo`
- `algorithm/model/Constant`

保留理由：

- 它们适合作为算法运行时模型；
- 未来可以作为 `converter` 的目标或来源对象；
- 不建议直接改造成数据库 Entity。

### 10.3 通用算法工具方法

建议保留并后续择机迁移位置：

- `utils.canReach`
- `utils.isSameEquipment`
- `Main.compareMapPaths`
- `Main.WriteMapCode`

保留理由：

- 这些方法具备独立业务价值；
- 可后续拆入 `domain service / diff service / visualization service`。

### 10.4 ZIP 导入导出中的纯工具逻辑

可局部保留：

- `ExampleServiceImpl.parseConfigurationFromFileName`
- `ExampleServiceImpl.checkConfiguration`
- `ExampleServiceImpl.packageZipFile`

保留前提：

- 作为导入导出适配层使用，而不是主存储实现。

## 11. 必须重构的代码

以下代码建议列为优先重构对象：

### 11.1 全部 Controller

包括：

- `AlgorithmController`
- `MapController`
- `NodeController`
- `PathController`
- `ConstraintController`
- `EquipmentController`
- `DirectoryController`
- `ExampleController`

原因：

- 当前 Controller 普遍过重；
- 直接依赖文件路径、目录、算法实例和响应拼装；
- 不符合后续 `/api + Result<T> + 权限校验 + 任务中心` 的目标。

### 11.2 `AlgorithmServiceImpl`

原因：

- 单例共享状态导致并发风险；
- 同时承担文件导入、流程转换、算法编排、临时文件管理、结果转换；
- 必须拆为：
  - 导入导出服务；
  - 算法执行服务；
  - 任务编排服务；
  - 可视化 / diff 服务。

### 11.3 所有直接改 YAML 的 CRUD Service

包括：

- `NodeServiceImpl`
- `PathServiceImpl`
- `ConstraintServiceImpl`
- `EquipmentServiceImpl`

原因：

- 这些实现全部基于文件系统和原始 Map 修改；
- 不适合迁移到 MyBatis-Plus；
- 必须替换为数据库驱动的领域服务。

### 11.4 `Main`

原因：

- 目前是杂糅式工具入口；
- 后续应拆分为：
  - `YamlConverter`
  - `AlgorithmModelConverter`
  - `MapDiffService`
  - `MapCodeRenderService`
  - 路径搜索工具类。

### 11.5 目录命名和文件命名驱动的查询逻辑

重点包括：

- `DirectoryController`
- `ExampleController`
- `ExampleServiceImpl` 中依赖文件名规则的部分

原因：

- 这类逻辑无法平滑支撑工作空间、用户隔离、任务中心、权限校验和结果查询。

## 12. 结论

当前仓库已经具备以下基础：

- 流程图结构编辑能力；
- YAML / JSON 导入；
- 三种优化算法；
- 优化结果输出；
- 基础的 ZIP 导入导出；
- 基础的 Mermaid `mapCode` 生成。

但整体仍然属于“单机文件驱动的流程优化工具后端”，而不是“数据库驱动的流程优化任务管理平台”。

后续重构的关键不是重写算法，而是完成以下四件事：

1. 用数据库接管流程图、节点、路径、装备、约束、任务、结果等主数据。
2. 用统一的 Service / Mapper / Converter 替代当前文件直写实现。
3. 用线程池任务中心替代当前同步优化请求。
4. 把 YAML / JSON / ZIP 降级为导入导出适配层，而不是主存储层。
