# 04_数据库与缓存策略

## 数据库设计原则

### 1. 数据库选型与版本

#### MySQL 8 特性利用
- **JSON支持**: 存储复杂结构化数据（如优化结果中的result_graph_json和diff_json）
- **窗口函数**: 复杂统计查询，如按时间窗口统计任务数量
- **CTE(公共表表达式)**: 复杂查询优化，提高查询可读性
- **默认字符集**: utf8mb4，支持完整Unicode（包括表情符号）

### 2. 表设计核心原则

#### 通用字段约定
所有业务主表（除operation_log外）都包含以下审计字段和逻辑删除字段：

```sql
-- 审计字段
created_by BIGINT,      -- 创建人用户ID
updated_by BIGINT,      -- 最后修改人用户ID
created_at DATETIME,    -- 创建时间，默认CURRENT_TIMESTAMP
updated_at DATETIME,    -- 更新时间，每次更新自动刷新

-- 逻辑删除字段  
deleted TINYINT(1) DEFAULT 0  -- 逻辑删除标志：0=未删除，1=已删除
```

**为什么采用逻辑删除？**
- **数据追溯**: 保留历史数据用于审计和分析，满足合规要求
- **关联数据完整性**: 避免外键约束失效，保持数据关联关系
- **恢复能力**: 误删后可快速恢复，无需从备份还原
- **查询优化**: 使用 `WHERE deleted = 0` 过滤删除记录，索引效率高

## 核心表结构详解

### 1. sys_user (用户表)

#### 字段设计详情
- **username**: VARCHAR(64)，用户名，要求唯一
- **password_hash**: VARCHAR(255)，使用BCrypt算法加密的密码哈希
- **nickname**: VARCHAR(64)，用户昵称，展示用
- **role_code**: VARCHAR(32)，角色代码，目前支持 'ADMIN' 和 'USER'
- **status**: TINYINT(1)，用户状态，1=启用，0=禁用
- **last_login_at**: DATETIME，记录最后登录时间，用于安全审计

#### 索引策略说明
```sql
-- 唯一索引考虑逻辑删除
CREATE UNIQUE INDEX uk_sys_user_username 
    ON sys_user(username, deleted);
-- 这样已删除用户的用户名可以被新用户使用
-- 但同一时间只有一个活跃用户可使用该用户名

-- 状态索引加速用户管理查询
CREATE INDEX idx_sys_user_status 
    ON sys_user(status);
-- 快速查询启用或禁用的用户列表
```

### 2. workspace (工作空间表)

#### 业务意义
工作空间是资源隔离的基本单元，每个用户可创建多个工作空间，但普通用户只能访问自己的工作空间。

#### 关键字段
- **owner_user_id**: BIGINT，工作空间所有者，建立用户与工作空间的关联
- **name**: VARCHAR(128)，工作空间名称，在同一用户下要求唯一
- **status**: TINYINT(1)，工作空间状态，1=正常，0=停用（软删除前状态）

#### 复合唯一索引设计
```sql
CREATE UNIQUE INDEX uk_workspace_owner_name 
    ON workspace(owner_user_id, name, deleted);
-- 确保同一用户下的工作空间名称不重复
-- 已删除的工作空间名称可以被复用
```

### 3. flow_graph (流程图主表)

#### 冗余字段设计
`total_time`、`total_precision`、`total_cost` 是冗余统计字段：

**冗余的好处**:
- **查询性能**: 列表查询时无需关联计算所有节点
- **排序优化**: 可基于统计值快速排序展示
- **缓存友好**: 统计值变化频率低，适合缓存

**冗余的代价**:
- **数据一致性**: 需要维护冗余字段与实际数据的一致性
- **更新开销**: 修改节点时需要重新计算并更新统计值

#### 状态字段设计
- **source_type**: 来源类型，'MANUAL'（手动创建）、'YAML_IMPORT'（YAML导入）、'OPTIMIZE_RESULT_RESTORE'（优化结果还原）
- **graph_status**: 流程图状态，'DRAFT'（草稿）、'READY'（就绪）、'ARCHIVED'（归档）

### 4. process_node (节点表)

#### 装备关联设计
节点通过 `equipment_id` 关联装备表，而不是直接存储装备名称：

**优势**:
- **数据一致性**: 装备信息变更时，所有关联节点自动更新
- **存储效率**: 避免重复存储相同的装备信息
- **查询灵活**: 可方便查询使用特定装备的所有节点

**实现细节**:
```java
// 转换时通过equipment_id查找装备名称
if (node.getEquipmentId() != null) {
    String equipmentName = equipments.stream()
        .filter(e -> e.getId().equals(node.getEquipmentId()))
        .findFirst()
        .map(EquipmentEntity::getName)
        .orElse(null);
    multiNode.setEquipmentName(equipmentName);
}
```

#### 排序字段设计
`sort_no` 字段用于前端展示排序，不承担算法拓扑排序职责：

**设计原则**:
- **展示与计算分离**: 展示顺序与算法计算顺序解耦
- **用户友好**: 允许用户自定义节点显示顺序
- **默认值**: 通常按创建时间或某种规则生成初始排序

### 5. process_path (路径表)

#### 防重复设计
同一流程图下禁止重复边（相同起点和终点）：

```sql
CREATE UNIQUE INDEX uk_process_path_graph_start_end 
    ON process_path(graph_id, start_node_id, end_node_id, deleted);
```

**业务意义**:
- **数据一致性**: 避免重复路径导致算法计算错误
- **存储优化**: 减少不必要的数据冗余
- **查询简化**: 确保路径查询结果的确定性

#### 关系类型预留
`relation_type` 字段目前只使用 'NORMAL'，但为未来扩展预留：
- **NORMAL**: 普通连接关系
- **CONSTRAINT**: 约束性连接
- **OPTIONAL**: 可选连接
- **PARALLEL**: 并行连接

### 6. optimize_task (优化任务表)

#### 任务编号生成策略
`task_no` 格式：`TASK{yyyyMMddHHmmss}{8位随机字符}`

**示例**: `TASK20240429123045A1B2C3D4`

**设计考虑**:
- **时间前缀**: 便于按时间排序和查询
- **随机后缀**: 防止时间冲突导致的重复
- **业务标识**: 'TASK'前缀明确业务含义
- **长度适中**: 20-30字符，适合存储和展示

#### 状态机设计
`task_status` 支持的状态流转：
```text
PENDING（待执行） → RUNNING（执行中） → SUCCESS（成功）
                                      ↓
                                    FAILED（失败） → (重试) → PENDING
```

**状态约束**:
- 仅FAILED状态允许重试
- 重试次数受 `max_retry_count` 限制
- SUCCESS状态不可重试

### 7. optimize_result (优化结果表)

#### JSON字段设计
使用JSON字段存储复杂结构数据：

**result_graph_json**: 优化后的完整流程图结构
**diff_json**: 优化前后的差异对比信息

**JSON存储的优势**:
- **灵活性**: 不需要预定义复杂的数据结构
- **开发效率**: 前端可直接使用，减少DTO转换
- **版本兼容**: 字段增减不影响数据库Schema

**JSON存储的劣势**:
- **查询限制**: 不能基于JSON内部字段高效查询
- **数据验证**: 需要在应用层验证JSON结构有效性
- **存储空间**: 可能比规范化存储占用更多空间

#### 统计字段精度
- **total_precision_before/after**: DECIMAL(10,4)，保留4位小数
- **score_ratio**: DECIMAL(12,6)，保留6位小数，保证计算精度

### 8. operation_log (操作日志表)

#### 日志脱敏策略
`request_params` 字段存储脱敏后的请求参数：

**需要脱敏的敏感信息**:
- 密码相关字段
- 认证Token
- 个人隐私信息

**脱敏示例**:
```json
原始参数: {"username": "admin", "password": "secret123"}
脱敏后: {"username": "admin", "password": "******"}
```

#### 性能字段
`cost_time_ms` 记录接口执行耗时，用于：
- **性能监控**: 识别慢接口
- **容量规划**: 评估系统负载
- **优化定位**: 定位性能瓶颈

## 外键关系设计

### 1. 逻辑外键 vs 物理外键

**项目选择逻辑外键的原因**:

**性能优势**:
- 避免外键检查带来的额外开销
- 批量导入数据时性能更好
- 分库分表时不需要维护外键关系

**灵活性优势**:
- 允许临时数据不一致（如数据迁移期间）
- 可先插入子表记录，再插入父表记录
- 数据清理更灵活，不受外键约束限制

**通过应用层保障一致性**:
- Service层事务管理
- 业务逻辑校验关联关系
- 定期数据一致性检查任务

### 2. 级联操作实现

#### 应用层级联删除
```java
@Transactional(rollbackFor = Exception.class)
public void deleteGraph(Long graphId) {
    // 1. 删除流程图主记录
    flowGraphMapper.deleteById(graphId);
    
    // 2. 删除所有关联节点
    processNodeMapper.delete(
        new LambdaQueryWrapper<ProcessNodeEntity>()
            .eq(ProcessNodeEntity::getGraphId, graphId)
    );
    
    // 3. 删除所有关联路径
    processPathMapper.delete(...);
    
    // 4. 删除所有关联约束
    constraintConditionMapper.delete(...);
    
    // 5. 删除所有关联装备
    equipmentMapper.delete(...);
    
    // 6. 清理相关缓存
    evictGraphCache(graphId);
}
```

**事务保障**: 整个操作在单个事务中，要么全部成功，要么全部回滚。

## 缓存策略设计

### 1. Redis缓存架构

#### 缓存层级设计
```
Level 1: 会话缓存 (satoken:login:token:*)
  ↓
Level 2: 任务状态缓存 (optimize:task:status:*)
  ↓  
Level 3: 优化结果缓存 (optimize:result:*)
  ↓
Level 4: (预留)流程图详情缓存 (graph:detail:*)
```

**分级设计的好处**:
- **热点分离**: 不同业务数据缓存策略不同
- **失效策略**: 各级缓存可设置不同的TTL
- **容量控制**: 可针对不同层级设置不同内存限制

#### 缓存Key设计规范
```java
// 业务前缀: 冒号分隔，便于Redis集群分片
private static final String TASK_STATUS_KEY_PREFIX = "optimize:task:status:";

// 完整Key示例: optimize:task:status:12345
private String taskStatusKey(Long taskId) {
    return TASK_STATUS_KEY_PREFIX + taskId;
}
```

**命名规范**:
1. 使用英文单词，全小写
2. 业务前缀明确标识缓存用途
3. 冒号分隔形成层次结构
4. 业务ID作为后缀

### 2. 缓存读写策略

#### 读缓存策略
```java
@Override
public OptimizeTaskVO getTask(Long taskId) {
    OptimizeTaskEntity task = getAccessibleTask(taskId);
    
    // 1. 优先从缓存读取状态
    String status = optimizeTaskCacheService.getTaskStatus(taskId);
    
    if (status == null) {
        // 2. 缓存未命中，回源数据库
        optimizeTaskCacheService.cacheTaskStatus(taskId, task.getTaskStatus());
    } else {
        // 3. 使用缓存状态（可能比数据库更新）
        task.setTaskStatus(status);
    }
    
    return OptimizeTaskConverter.toTaskVO(task);
}
```

**缓存优先原则**:
- 缓存可能存在比数据库更新的状态（如正在运行的任务）
- 缓存未命中时查询数据库并回填
- 缓存数据可能短暂不一致，但最终一致

#### 写缓存策略
```java
@Override
@Transactional(rollbackFor = Exception.class)
public OptimizeTaskSubmitVO submitTask(CreateOptimizeTaskRequest request) {
    // 1. 创建数据库记录
    getOptimizeTaskMapper().insert(task);
    
    // 2. 立即写入缓存（事务内）
    optimizeTaskCacheService.cacheTaskStatus(task.getId(), OptimizeTaskStatus.PENDING);
    
    // 3. 异步提交任务
    submitAfterCommit(task.getId());
    
    return OptimizeTaskConverter.toSubmitVO(task);
}
```

**双写策略**:
- 数据库和缓存都在事务内更新
- 缓存更新失败不影响事务提交
- 缓存数据可能比数据库稍旧，但很快会同步

### 3. 缓存一致性保障

#### Cache-Aside模式实现
```text
读操作流程:
  1. 检查缓存是否存在
  2. 缓存存在 → 返回缓存数据
  3. 缓存不存在 → 查询数据库 → 写入缓存 → 返回数据

写操作流程:
  1. 更新数据库
  2. 删除缓存（或更新缓存）
  3. 后续读操作会重新加载最新数据
```

**为什么不先删除缓存再更新数据库？**
如果先删缓存，在数据库更新完成前，可能有其他线程：
1. 发现缓存不存在
2. 读取旧数据库数据
3. 将旧数据写入缓存
导致缓存中一直是旧数据。

#### 最终一致性方案
**方案1：先更新数据库，再删除缓存**
- 优点：实现简单，数据安全优先
- 缺点：可能存在短暂的数据不一致

**方案2：使用双删策略**
```java
// 1. 先删除缓存
cache.delete(key);

// 2. 更新数据库
db.update(data);

// 3. 延迟再次删除缓存（处理并发读导致的脏数据）
asyncDelayDelete(key, 1000); // 1秒后再次删除
```

**方案3：使用消息队列保证顺序**
- 优点：强一致性保证
- 缺点：系统复杂度高，性能开销大

**项目选择方案1的原因**:
- 业务对短暂不一致容忍度高
- 实现简单，维护成本低
- 性能影响最小

### 4. 缓存降级与熔断

#### RedisSafeClient设计
```java
@Component
public class RedisSafeClient {
    private final RedisTemplate<String, Object> redisTemplate;
    
    public void set(String key, Object value, long timeout, TimeUnit unit) {
        try {
            redisTemplate.opsForValue().set(key, value, timeout, unit);
        } catch (Exception ex) {
            // 只记录日志，不抛出异常
            log.warn("Redis写入失败: key={}, error={}", key, ex.getMessage());
        }
    }
    
    public Object get(String key) {
        try {
            return redisTemplate.opsForValue().get(key);
        } catch (Exception ex) {
            log.warn("Redis读取失败: key={}, error={}", key, ex.getMessage());
            return null;  // 返回null触发回源
        }
    }
}
```

**降级策略**:
- **读降级**: Redis异常时直接查询数据库
- **写降级**: Redis异常时只记录日志，不中断业务流程
- **监控告警**: 记录Redis异常频率，超过阈值告警

#### 熔断机制考虑
虽然当前未实现，但可考虑引入：
- **失败计数**: 连续失败N次后开启熔断
- **熔断状态**: 熔断期间直接走降级逻辑
- **半开状态**: 定期尝试恢复，成功后关闭熔断

### 5. 缓存监控与优化

#### 监控指标
```yaml
# 需要监控的缓存指标
cache:
  metrics:
    - hit_rate: 缓存命中率
    - miss_rate: 缓存未命中率
    - eviction_count: 缓存驱逐次数
    - memory_usage: 内存使用率
    - command_latency: 命令延迟
```

#### 优化策略
**根据命中率调整TTL**:
- 命中率 > 90%: 可适当增加TTL
- 命中率 < 60%: 考虑减少TTL或调整缓存策略
- 命中率 < 30%: 可能需要重新评估缓存必要性

**根据内存使用调整策略**:
- 内存使用 > 80%: 考虑使用LRU淘汰策略
- 内存使用持续增长: 检查是否有内存泄漏
- 大Key优化: 拆分大Value为多个小Key

## 性能优化策略

### 1. 数据库查询优化

#### 索引覆盖查询优化
```sql
-- 优化前：需要回表查询所有字段
SELECT * FROM flow_graph WHERE workspace_id = 1;

-- 优化后：使用覆盖索引，避免回表
SELECT id, name, total_time 
FROM flow_graph 
WHERE workspace_id = 1;
-- 如果workspace_id上有索引，且查询字段都在索引中，则不需要回表
```

#### 分页查询优化
**深分页问题**:
```sql
-- 性能差：OFFSET越大性能越差
SELECT * FROM optimize_task 
ORDER BY created_at DESC 
LIMIT 20 OFFSET 10000;
-- MySQL需要扫描10020行，然后丢弃前10000行
```

**游标分页优化**:
```sql
-- 基于ID的游标分页
SELECT * FROM optimize_task 
WHERE id < ?  -- 上一页最后一条记录的ID
ORDER BY id DESC 
LIMIT 20;

-- 基于时间的游标分页  
SELECT * FROM optimize_task
WHERE created_at < ?  -- 上一页最后一条记录的时间
ORDER BY created_at DESC
LIMIT 20;
```

#### 批量操作优化
```java
// MyBatis-Plus批量插入优化
boolean success = optimizeResultMapper.insertBatchSomeColumn(resultList);

// 分批处理避免超大事务
int batchSize = 1000;
for (int i = 0; i < total; i += batchSize) {
    List<OptimizeResultEntity> batch = resultList.subList(i, Math.min(i + batchSize, total));
    optimizeResultMapper.insertBatchSomeColumn(batch);
}
```

### 2. 连接池配置优化

#### HikariCP推荐配置
```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20           # CPU核心数 × 2 + 有效磁盘数
      minimum-idle: 5                 # 保持的最小空闲连接数
      connection-timeout: 30000       # 连接超时时间（毫秒）
      idle-timeout: 600000            # 连接空闲超时时间（毫秒）
      max-lifetime: 1800000           # 连接最大生命周期（毫秒）
      connection-test-query: SELECT 1 # 连接测试查询语句
      
      # 连接泄漏检测
      leak-detection-threshold: 60000 # 连接泄漏检测阈值（毫秒）
```

**配置原则**:
- **maximum-pool-size**: 不宜过大，避免连接竞争
- **minimum-idle**: 不宜过小，避免频繁创建连接
- **idle-timeout**: 适当设置，释放长时间空闲连接
- **max-lifetime**: 定期更换连接，避免长时间连接的问题

## 数据迁移与备份

### 1. 渐进式数据迁移策略

#### 三个阶段迁移
```text
阶段1: 双写模式（当前阶段）
  - 新数据写入MySQL
  - 旧数据保留在YAML文件
  - 提供YAML导入MySQL工具
  - 新接口使用MySQL，旧接口使用YAML

阶段2: 数据同步
  - 将常用YAML文件批量导入MySQL
  - 验证数据一致性
  - 提供数据对比工具

阶段3: 完全切换
```

```bash
#!/bin/bash
# 数据库备份脚本
BACKUP_DIR="/backup/mysql"

  --single-transaction \
  --routines \
  --triggers \
  --events \
```
