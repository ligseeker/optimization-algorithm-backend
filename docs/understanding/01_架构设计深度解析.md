# 01_架构设计深度解析

## 项目架构概览

本项目采用**渐进式重构**策略，在保留原有算法能力的基础上，将文件驱动的单机应用升级为**基于Spring Boot的分布式架构平台**。核心目标是解决数据一致性、并发扩展、权限控制和任务追踪等问题。

### 分层架构设计

```
┌─────────────────────────────────────────────┐
│               Controller层                   │
│   • RESTful API接口                          │
│   • 参数校验与转换                            │

│   • Sa-Token注解鉴权                         │
└─────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────┐
│             AppService层 (业务逻辑层)        │
│   • 业务逻辑编排                              │
│   • 事务边界管理                              │
│   • 资源权限校验                              │
│   • 业务异常处理                              │
└─────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────┐
│        PersistenceService层 (持久化层)       │
│   • 数据访问抽象                              │

```

## 核心设计模式应用

### 1. 依赖注入与接口隔离
- **ObjectProvider模式**: 用于优雅处理数据库未配置情况
- **服务接口隔离**: 每层只依赖抽象接口，不依赖具体实现
- **构造器注入**: 所有依赖通过构造器注入，保证不可变性和测试性

### 2. 策略模式应用
```java
// 算法执行器的策略选择
switch (task.getAlgorithmType()) {
    case 1: algorithm1.optimize(map1, mode); break;
    case 2: algorithm2.optimize(map1, factors, mode); break;
    case 3: algorithm3.optimize(map1, factors, mode); break;
}
```

### 3. 门面模式应用
- **ResourceAccessService**: 统一资源访问校验门面
- **OptimizeTaskAppService**: 优化任务全链路编排门面

### 4. 观察者模式应用
```java
// 事务提交后异步执行任务
TransactionSynchronizationManager.registerSynchronization(
    new TransactionSynchronization() {
        @Override
        public void afterCommit() {
            algorithmExecutor.submit(taskId);
        }
    }
);
```

## 架构演进决策

### 1. 为什么选择渐进式重构？
- **风险可控**: 保留旧算法能力，可随时回滚
- **平滑迁移**: 新老接口并存，逐步切换
- **团队适应**: 渐进学习新技术栈

### 2. 为什么将YAML降级为交换格式？
- **事务一致性**: YAML文件无法保证ACID
- **并发控制**: 文件锁机制复杂且性能差
- **版本管理**: 数据库可轻松实现版本化和审计
- **权限控制**: 数据库可精细控制访问权限

### 3. 为什么采用ObjectProvider处理数据库依赖？
```java
// 优雅降级设计
private SysUserMapper getSysUserMapper() {
    SysUserMapper mapper = sysUserMapperProvider.getIfAvailable();
    if (mapper == null) {
        throw new BusinessException(ErrorCode.SYSTEM_ERROR, "数据库未配置或不可用");
    }
    return mapper;
}
```

**优势**:
- 允许应用在数据库未配置时部分功能可用
- 避免应用启动失败，支持功能逐步上线
- 便于单元测试mock

## 分层职责清晰界定

### Controller层 (接口层)
- **职责**: HTTP协议适配、参数校验、响应格式化
- **禁止**: 业务逻辑、数据库操作、算法调用
- **示例**: `AuthController`, `OptimizeTaskController`

### AppService层 (业务逻辑层)
- **职责**: 业务流程编排、事务管理、权限校验
- **特征**: 聚合多个PersistenceService，实现完整业务用例
- **示例**: `OptimizeTaskAppServiceImpl`

### PersistenceService层 (持久化层)
- **职责**: 数据存取抽象、缓存集成、异常转换
- **特征**: 每个Entity对应一个Service，职责单一
- **示例**: `SysUserPersistenceService`, `OptimizeTaskPersistenceService`

### Converter层 (转换层)
- **职责**: 数据类型转换、数据脱敏、格式适配
- **关键**: `ProcessMapConverter`处理算法模型与数据库模型的转换

## 事务边界设计

### 1. 事务类型选择
- **声明式事务**: `@Transactional`注解管理
- **编程式事务**: 复杂业务场景使用

### 2. 事务传播策略
```java
@Transactional(rollbackFor = Exception.class)
public OptimizeTaskSubmitVO submitTask(CreateOptimizeTaskRequest request) {
    // 1. 校验与数据准备（非事务）
    // 2. 创建任务记录（数据库事务）
    // 3. 提交异步任务（事务提交后执行）
}
```

### 3. 事务一致性保证
- **创建任务**: 原子性创建记录和缓存状态
- **删除资源**: 级联删除相关子资源
- **导入流程**: 解析、校验、转换、写入全程事务

- **错误码分层**: 按模块和错误类型编码

## 配置与扩展性设计

### 1. 配置外部化
- **application.yml**: 主配置文件
- **app.cache**: 缓存相关配置（TTL、key前缀）
- **app.threadpool**: 线程池配置

### 2. 扩展点预留
- **算法扩展**: 策略模式支持新算法类型
- **存储扩展**: 可切换不同的数据库或缓存
- **认证扩展**: Sa-Token支持多种认证方式

### 3. 健康检查与监控
- **Spring Boot Actuator**: 应用健康状态
- **自定义指标**: 任务执行统计、缓存命中率
- **操作日志**: 关键操作审计追踪

## 重构过程中的关键决策

### 1. 不将算法模型直接映射为Entity
**原因**:
- 算法模型频繁变化，数据库Schema应保持稳定
- 算法模型包含计算逻辑，不适合持久化
- 分离业务领域与算法领域，降低耦合

**实现**:
```java
// Converter负责双向转换
ProcessMap processMap = ProcessMapConverter.toProcessMap(
    graph, nodes, paths, constraints, equipments
);
```

### 2. 资源权限校验链设计
```text
nodeId → graphId → workspaceId → ownerUserId → 当前用户
```

**优势**:
- 统一权限校验逻辑
- 避免权限绕过漏洞
- 便于权限控制扩展

### 3. 任务状态机的缓存策略
- **MySQL**: 最终状态存储
- **Redis**: 实时状态缓存（TTL: 24小时）
- **回源机制**: 缓存未命中时从数据库加载

## 总结

本项目架构设计体现了以下核心原则：

1. **渐进式演进**: 在不破坏现有功能的前提下逐步升级
2. **关注点分离**: 清晰的层级边界和模块划分
3. **扩展性优先**: 预留扩展点，支持未来发展
4. **稳定性保障**: 优雅降级、事务一致性、错误处理
5. **可维护性**: 代码结构清晰，易于理解和修改

这种架构设计不仅解决了原有系统的痛点，还为未来的功能扩展和技术演进奠定了坚实基础。
