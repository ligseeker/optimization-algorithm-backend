# 02_核心技术实现详解

## Sa-Token 认证与鉴权实现

### 1. 登录流程详解

#### 登录核心逻辑
位于 `AuthServiceImpl.java` 的核心登录流程包含以下关键步骤：

**步骤1 - 用户查询**: 根据用户名查询数据库，获取用户实体

**步骤2 - 密码校验**: 使用 BCryptPasswordEncoder 验证密码哈希
```java
if (!PasswordUtils.matches(request.getPassword(), user.getPasswordHash())) {
    throw new BusinessException(ErrorCode.UNAUTHORIZED, "用户名或密码错误");
}
```

**步骤3 - 状态校验**: 验证用户是否被禁用（status字段）

**步骤4 - Sa-Token登录**: 调用 `StpUtil.login(user.getId())` 创建会话

**步骤5 - 更新登录时间**: 更新 `last_login_at` 字段（LocalDateTime类型）

**步骤6 - 返回Token**: 返回包含 tokenName 和 tokenValue 的响应

#### 密码加密策略
`PasswordUtils.java` 使用 BCryptPasswordEncoder 实现：
- **BCrypt算法**: 自带salt，每次加密结果不同，防止彩虹表攻击
- **强度因子**: 默认10，可配置调整计算强度
- **安全性**: 抗暴力破解，单次校验耗时约100ms

### 2. Token与会话管理

#### Token存储机制
- **Redis存储**: Sa-Token默认将Token存储在Redis中
- **Key格式**: `satoken:login:token:{tokenValue}`
- **Value内容**: 用户ID、登录时间、设备信息等
- **TTL配置**: `sa-token.timeout=7200` (2小时)

#### Token验证流程
**注解式验证**: Controller层使用 `@SaCheckLogin` 注解
```java
@SaCheckLogin
public Result<CurrentUserVO> getCurrentUser() {
    return Result.success(authService.getCurrentUser());
}
```

**主动验证**: Service层调用 `StpUtil.checkLogin()` 验证登录状态

#### Token续签策略
- **自动续期**: 每次活跃操作自动延长Token有效期
- **强制过期**: `logout()` 方法立即清除Redis中的Token

### 3. 角色权限控制

#### 角色定义
系统定义了两个角色：
- **ADMIN**: 管理员角色，可访问所有资源
- **USER**: 普通用户角色，只能访问自己的资源

#### 角色校验实现
`SaTokenRoleProvider.java` 实现 StpInterface 接口：
```java
@Override
public List<String> getRoleList(Object loginId, String loginType) {
    SysUserEntity user = sysUserMapper.selectById(loginId);
    return Collections.singletonList(user.getRoleCode());
}
```

#### 权限隔离逻辑
`ResourceAccessServiceImpl.java` 实现资源访问校验：
- **管理员权限**: 可访问所有工作空间和流程图
- **普通用户权限**: 只能访问 `owner_user_id` 等于当前用户ID的资源
- **校验链路**: nodeId → graphId → workspaceId → ownerUserId → 当前用户

### 4. 会话安全机制

#### 并发登录控制
- **默认策略**: 允许同一账号多设备登录
- **Token隔离**: 不同设备生成不同Token
- **会话查询**: `StpUtil.getTokenValueListByLoginId(userId)` 获取所有Token

#### 登录设备识别
- **设备标识**: 通过 User-Agent 或自定义设备ID
- **设备管理**: 可实现踢出特定设备功能

## 异步任务执行机制

### 1. 线程池配置

#### ThreadPoolTaskExecutor配置详解
`ThreadPoolConfig.java` 定义了专门的优化任务线程池：

**核心参数**:
- **corePoolSize**: 4 - 核心线程数，始终保持活跃
- **maxPoolSize**: 8 - 最大线程数，任务高峰时临时扩展
- **queueCapacity**: 100 - 任务队列容量，防止任务无限堆积
- **keepAliveSeconds**: 60 - 非核心线程空闲超时时间

**线程命名**: `optimize-task-{序号}`，便于日志追踪和监控

**拒绝策略**: CallerRunsPolicy - 队列满时由调用线程执行，避免任务丢失

**设计要点**:
- 有界队列防止OOM
- CallerRunsPolicy保证任务不丢失
- 核心线程超时释放资源

### 2. 任务提交流程

#### 事务后提交设计
`OptimizeTaskAppServiceImpl.java` 的 `submitTask` 方法：

**关键设计**: 使用 `TransactionSynchronizationManager` 确保任务在事务提交后执行
```java
private void submitAfterCommit(Long taskId) {
    if (TransactionSynchronizationManager.isActualTransactionActive()) {
        TransactionSynchronizationManager.registerSynchronization(
            new TransactionSynchronization() {
                @Override
                public void afterCommit() {
                    algorithmExecutor.submit(taskId);
                }
            }
        );
        return;
    }
    algorithmExecutor.submit(taskId);  // 非事务场景直接提交
}
```

**为什么要事务后提交**:
- 确保任务记录已持久化到MySQL
- 防止事务回滚导致任务执行但记录不存在
- 避免数据库查询不到任务状态

#### 任务编号生成策略
`buildTaskNo()` 方法生成唯一任务编号：
- **格式**: `TASK{yyyyMMddHHmmss}{8位随机UUID}`
- **示例**: `TASK20260427170530A1B2C3D4`
- **唯一性保证**: 时间戳+随机数组合

### 3. 任务执行流程

#### AlgorithmExecutor核心逻辑
`AlgorithmExecutorImpl.java` 实现完整的任务执行链路：

**阶段1 - 任务检索**: 从数据库加载任务实体

**阶段2 - 状态检查**: 验证任务是否处于PENDING或RUNNING状态

**阶段3 - 状态标记**: 调用 `markRunning()` 更新状态

**阶段4 - 构建上下文**: 调用 `buildContext()` 加载流程图数据

**阶段5 - 执行算法**: 调用 `runAlgorithm()` 执行优化计算

**阶段6 - 持久化结果**: 成功时调用 `persistSuccess()`，失败时调用 `persistFailure()`

#### 状态流转机制
`OptimizeTaskStateServiceImpl.java` 实现状态状态机：

**状态定义**:
- **PENDING**: 待执行，任务刚创建
- **RUNNING**: 正在执行，算法计算中
- **SUCCESS**: 成功完成，结果已持久化
- **FAILED**: 执行失败，记录错误信息

**状态转换**:
```text
PENDING → RUNNING → SUCCESS
                  ↓
                FAILED → (重试) → PENDING
```

**状态更新流程**:
```java
public void markRunning(Long taskId) {
    // 1. 更新数据库
    OptimizeTaskEntity update = new OptimizeTaskEntity();
    update.setTaskStatus(OptimizeTaskStatus.RUNNING);
    update.setStartedAt(LocalDateTime.now());
    optimizeTaskMapper.updateById(update);
    
    // 2. 更新缓存
    optimizeTaskCacheService.cacheTaskStatus(taskId, OptimizeTaskStatus.RUNNING);
}
```

### 4. 任务重试机制

#### 重试逻辑实现
`retryTask()` 方法实现有限次数重试：

**重试条件**:
- **状态限制**: 仅FAILED状态可重试
- **次数限制**: `retryCount < maxRetryCount`（默认最多3次）

**重试流程**:
```text
1. 校验任务状态和重试次数
2. 清理旧缓存（状态缓存和结果缓存）
3. 更新状态为PENDING，增加重试计数
4. 提交到线程池重新执行
```

**幂等性保证**:
- 复用原任务ID和参数
- 不创建新任务记录
- 重试次数累加，达到上限后禁止重试

## Redis 缓存策略实现

### 1. RedisSafeClient封装

#### 安全操作封装理念
`RedisSafeClient.java` 实现了异常降级机制：

**核心原则**:
- **Redis异常不影响主业务**: 所有操作捕获异常并记录日志
- **降级到数据库**: 缓存失败时直接查询MySQL
- **异步日志**: 使用warn级别记录Redis操作失败

**实现示例**:
```java
public Object get(String key) {
    try {
        return redisTemplate.opsForValue().get(key);
    } catch (Exception ex) {
        log.warn("Redis读取失败: key={}, error={}", key, ex.getMessage());
        return null;  // 返回null触发回源数据库
    }
}
```

### 2. 任务状态缓存

#### 状态缓存Key设计
`OptimizeTaskCacheServiceImpl.java` 定义缓存Key格式：

**Key前缀**: `optimize:task:status:{taskId}`
**Value类型**: String（状态枚举名称）
**TTL配置**: 默认1440分钟（24小时）

**缓存策略**:
- **写入时机**: 状态变更时立即写入
- **读取优先**: 查询任务状态时优先读Redis
- **回源兜底**: 缓存未命中或异常时查询MySQL
- **失效策略**: 任务状态最终稳定后可过期

### 3. 优化结果缓存

#### 结果缓存设计
**Key前缀**: `optimize:result:{taskId}`
**Value类型**: OptimizeResultVO对象
**TTL配置**: 默认1440分钟（24小时）

**缓存时机**:
- 算法执行成功后立即缓存
- 包含完整结果数据（指标、Diff、MapCode）

**失效策略**:
- 任务重试时清理旧结果缓存
- 用户主动查询时更新缓存

### 4. 缓存一致性保障

#### Cache-Aside模式应用

**写操作流程**:
```text
1. 更新MySQL数据库
2. 删除Redis缓存（异步执行更安全）
3. 下次查询时缓存会重建
```

**读操作流程**:
```text
1. 尝试从Redis读取
2. 缓存未命中则查询MySQL
3. 异步回填Redis缓存
```

**一致性原则**:
- **数据库为主**: MySQL是最终数据源
- **缓存为辅**: Redis只用于性能加速
- **不强依赖**: Redis异常不影响业务正确性

## 事务边界管理

### 1. 声明式事务应用

#### 基本事务配置
使用 `@Transactional` 注解管理事务：
- **rollbackFor**: Exception.class，任何异常都回滚
- **propagation**: 默认REQUIRED，加入现有事务
- **isolation**: 默认READ_COMMITTED，平衡并发与一致性

#### 事务传播场景
**REQUIRED模式**: 最常用的传播模式
```java
@Transactional
public void methodA() {
    // 主事务
    methodB();  // 加入主事务
}

@Transactional
public void methodB() {
    // 加入methodA的事务
}
```

**REQUIRES_NEW模式**: 独立事务
```java
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void logOperation() {
    // 独立事务，主事务回滚不影响日志
}
```

### 2. 复杂事务场景

#### 级联删除事务
删除流程图时需要级联删除所有关联资源：
```text
事务开始
  ↓
删除 flow_graph
  ↓
删除 process_node (多条)
  ↓
删除 process_path (多条)
  ↓
删除 constraint_condition (多条)
  ↓
删除 equipment (多条)
  ↓
清理缓存
  ↓
事务提交
```

**事务保证**: 全部成功或全部回滚，避免部分删除导致数据不一致

### 3. 事务隔离级别选择

#### 隔离级别应用场景
- **READ_COMMITTED**: 普通CRUD操作，避免脏读
- **REPEATABLE_READ**: 高并发计数更新，防止幻读
- **SERIALIZABLE**: 分布式事务场景，最严格隔离

#### MySQL默认隔离级别
MySQL默认使用 REPEATABLE_READ，适合大多数场景：
- 保证同一事务中多次读取结果一致
- 防止幻读（Phantom Read）
- 性能损失可接受

### 4. 事务超时控制

#### 超时配置
```java
@Transactional(timeout = 30)  // 30秒超时
public void longRunningOperation() {
    // 长时间操作
}
```

**超时处理**: 事务执行超过指定时间后自动回滚，防止长时间锁定资源
