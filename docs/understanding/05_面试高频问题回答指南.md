### Q2: 为什么选择渐进式重构而不是重写？

**回答要点**:
1. **风险控制**: 保留旧算法能力，可随时回滚，降低重构风险
2. **平滑迁移**: 新老接口并存，逐步切换，不影响现有用户
3. **团队适应**: 渐进学习新技术栈，降低团队学习成本
4. **业务连续性**: 保证业务不中断，重构过程透明

**具体措施**:
- 保留algorithm包核心算法逻辑
- 新增/api前缀的新接口，旧接口暂时保留
- YAML作为导入导出格式，逐步迁移到MySQL主存储
- 分阶段实施：数据库设计 → 持久化层 → 业务逻辑 → 前端集成

### Q3: 项目中最复杂的技术挑战是什么？如何解决的？

**回答示例**:

**技术细节**:
- 使用TransactionSynchronization确保事务提交后才提交任务

## 架构设计类问题

### Q4: 项目的分层架构是怎样的？各层职责是什么？

**回答结构**:
```
┌─────────────────┐
│   Controller层  │ ← HTTP协议适配、参数校验、统一响应
├─────────────────┤
│   AppService层  │ ← 业务逻辑编排、事务管理、权限校验
├─────────────────┤
│ Persistence层   │ ← 数据访问抽象、MyBatis-Plus封装
├─────────────────┤
│   Converter层   │ ← 数据类型转换、算法模型↔业务模型转换
└─────────────────┘
```

**各层职责详解**:
1. **Controller**: 只负责HTTP交互，禁止包含业务逻辑
2. **AppService**: 聚合多个PersistenceService，实现完整业务用例
3. **Persistence**: 每个Entity对应一个Service，职责单一
4. **Converter**: 关键的是ProcessMapConverter，处理算法模型转换

**设计原则**:
- 上层依赖下层抽象，不依赖具体实现
- 每层有明确的责任边界
- 使用构造器注入，保证不可变性

### Q5: 为什么将YAML降级为交换格式而不是主存储？

**对比分析**:
| 方面 | YAML作为主存储 | YAML作为交换格式 + MySQL主存储 |
|------|---------------|-------------------------------|
| 事务一致性 | 无法保证ACID | 完整的事务支持 |
| 并发控制 | 文件锁复杂，性能差 | 数据库行级锁，性能好 |
| 权限控制 | 文件系统权限粗粒度 | 数据库行级权限控制 |
| 查询能力 | 需要加载整个文件 | SQL灵活查询，支持索引 |
| 扩展性 | 单机限制 | 支持读写分离、分库分表 |
| 数据恢复 | 备份恢复复杂 | 完善的备份恢复机制 |

**结论**: YAML适合作为数据交换格式，但不适合作为企业级应用的主数据存储。

### Q6: 为什么采用ObjectProvider处理数据库依赖？

**回答要点**:
1. **优雅降级**: 允许应用在数据库未配置时部分功能可用
2. **渐进上线**: 避免应用启动失败，支持功能逐步上线
3. **测试友好**: 便于单元测试mock数据库依赖
4. **配置灵活**: 支持不同环境的不同数据库配置

**代码示例**:
```java
private SysUserMapper getSysUserMapper() {
    SysUserMapper mapper = sysUserMapperProvider.getIfAvailable();
    if (mapper == null) {
        throw new BusinessException(ErrorCode.SYSTEM_ERROR, "数据库未配置");
    }
    return mapper;
}
```

**应用场景**:
- 开发环境可能不配置数据库
- 某些功能模块可独立于数据库运行
- 数据库迁移或维护期间可降级运行

## 技术实现类问题

### Q7: 如何实现异步任务执行？事务如何处理？

**执行流程**:
```text
1. 用户提交任务 → Controller接收请求
2. Service创建任务记录（数据库事务）
3. 事务提交后提交到线程池（TransactionSynchronization.afterCommit）
4. 线程池异步执行算法
5. 更新任务状态（SUCCESS/FAILED）
```

**事务处理关键**:
```java
// 确保任务记录已持久化后才执行算法
TransactionSynchronizationManager.registerSynchronization(
    new TransactionSynchronization() {
        @Override
        public void afterCommit() {
            algorithmExecutor.submit(taskId); // 事务成功后才提交
        }
    }
);
```

**设计考虑**:
- 防止事务回滚但任务已执行的情况
- 避免任务执行但数据库记录不存在的问题
- 保证任务状态的可追踪性

### Q8: 任务状态机是如何设计的？如何保证状态一致性？

**状态定义**:
```java
public class OptimizeTaskStatus {
    public static final String PENDING = "PENDING";   // 待执行
    public static final String RUNNING = "RUNNING";   // 执行中  
    public static final String SUCCESS = "SUCCESS";   // 成功
    public static final String FAILED = "FAILED";     // 失败
}
```

**状态流转**:
```text
PENDING → RUNNING → SUCCESS
                  ↓
                FAILED → (重试) → PENDING
```

**一致性保障**:
1. **数据库最终存储**: MySQL存储最终状态
2. **Redis实时缓存**: 缓存当前状态，TTL 24小时
3. **状态更新原子性**: 使用updateById保证更新原子性
4. **缓存同步**: 状态变更时同步更新Redis

**重试机制**:
- 仅FAILED状态允许重试
- 重试次数上限控制（默认3次）
- 重试时清理旧缓存，避免脏数据

### Q9: Redis缓存策略是怎样的？如何保证缓存一致性？

**缓存架构**:
```
Level 1: 任务状态缓存 (optimize:task:status:{taskId})
Level 2: 优化结果缓存 (optimize:result:{taskId})
Level 3: 会话缓存 (satoken:login:token:{tokenValue})
```

**一致性策略**:
**Cache-Aside模式**:
```text
读: 先读缓存 → 未命中 → 读数据库 → 回填缓存
写: 更新数据库 → 删除缓存 → 下次读时重建
```

**为什么先更新数据库再删除缓存？**
- 数据安全优先：保证数据库数据正确性
- 短暂不一致可接受：缓存脏读在可接受范围内
- 实现简单：不需要复杂的分布式事务

**降级策略**:
```java
public class RedisSafeClient {
    public Object get(String key) {
        try {
            return redisTemplate.opsForValue().get(key);
        } catch (Exception ex) {
            log.warn("Redis读取失败", ex);
            return null;  // 触发回源数据库
        }
    }
}
```

### Q10: 如何设计资源权限校验链？

**校验链设计**:
```text

```

```java
public FlowGraphEntity getAccessibleGraph(Long graphId) {
    // 1. 查询流程图
    FlowGraphEntity graph = flowGraphMapper.selectById(graphId);
    
    // 2. 校验工作空间权限
    getAccessibleWorkspace(graph.getWorkspaceId());
    
    return graph;

public WorkspaceEntity getAccessibleWorkspace(Long workspaceId) {
    WorkspaceEntity workspace = workspaceMapper.selectById(workspaceId);
    
    // 管理员可访问所有，普通用户只能访问自己的
    if (!currentUserService.isAdmin()) {
        if (!workspace.getOwnerUserId().equals(currentUserService.getCurrentUserId())) {
            throw new BusinessException(ErrorCode.FORBIDDEN, "无权限");
        }
    }
    
    return workspace;
}
```

**设计优势**:
- 统一权限校验逻辑
- 避免权限绕过漏洞
- 便于权限控制扩展

```java
// 数据库实体 → 算法模型
ProcessMap processMap = ProcessMapConverter.toProcessMap(

Map<String, Object> diff = buildSimplifiedDiff(pathDiff, metrics);
```

- 转换器处理数据格式差异和关联关系

### Q13: 综合评分是如何计算的？

**评分公式**:
```
综合评分 = (时间改进率 × 时间权重 + 精度改进量 × 精度权重 + 成本改进率 × 成本权重) / 总权重
```

**具体计算**:
```java
double timePart = beforeTime <= 0 ? 0 : (beforeTime - afterTime) / (double)beforeTime;
double precisionPart = afterPrecision - beforePrecision;  // 精度越大越好
double costPart = beforeCost <= 0 ? 0 : (beforeCost - afterCost) / (double)beforeCost;

// 权重归一化
double wTime = (double)factors[0] / totalWeight;
double wPrecision = (double)factors[1] / totalWeight;
double wCost = (double)factors[2] / totalWeight;

double ratio = timePart * wTime + precisionPart * wPrecision + costPart * wCost;
```

**评分意义**:
- **正数**: 优化效果正面
- **负数**: 优化效果负面  
- **零**: 无明显变化
- **范围**: 通常[-1, 1]，根据权重可超出
