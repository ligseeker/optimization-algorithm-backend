# 02_工作空间模块

## 一、模块定位

工作空间是资源归属链的起点。它回答的问题是：

- 当前用户拥有哪些流程图容器
- 某个 graph 属于哪个工作空间
- 普通用户和管理员能看到哪些工作空间

## 二、模块级调用链

```text
WorkspaceController
  -> WorkspaceAppServiceImpl
  -> CurrentUserServiceImpl
  -> WorkspaceMapper
  -> WorkspaceConverter
  -> WorkspaceVO / PageResult<WorkspaceVO>
```

异常主线：

- workspace 不存在 -> `RESOURCE_NOT_FOUND`
- 无权限访问 -> `FORBIDDEN`
- 名称重复 -> `CONFLICT`
- 数据库不可用 -> `SYSTEM_ERROR`

## 三、关键文件逐个说明

### 1. `module/workspace/controller/WorkspaceController.java`

职责：

- 新建工作空间
- 分页查询
- 查询详情
- 更新
- 删除

你读它时重点看：

- 路径是 `/api/workspaces`
- 每个方法都非常薄
- `@OperationLog` 只标记关键写操作

它说明重构后 Controller 的设计目标是：

- 只收参数
- 不直接写 SQL
- 不直接做权限判断

### 2. `module/workspace/service/impl/WorkspaceAppServiceImpl.java`

这是工作空间模块真正的业务中心。

#### 字段逐项说明

- `workspaceMapperProvider`
  直接访问工作空间表
- `currentUserService`
  决定当前用户是谁、是不是 admin

#### 主要方法逐项说明

- `createWorkspace`
  1. 取当前用户
  2. 校验当前用户下工作空间名称唯一
  3. 组装 `WorkspaceEntity`
  4. 写库
  5. 返回 `WorkspaceVO`

- `listWorkspaces`
  这里最能体现权限差异：
  - admin：查全部
  - 普通用户：只查自己 `ownerUserId` 下的记录

- `getWorkspace`
  依赖私有方法 `getAccessibleWorkspace`

- `updateWorkspace`
  先校验可访问，再校验名称冲突，再更新

- `deleteWorkspace`
  这里只删工作空间本身，是否需要级联资源清理要结合后续 graph 设计一起理解

- `getAccessibleWorkspace`
  这是该模块自己的资源权限入口，和通用 `ResourceAccessServiceImpl` 逻辑非常相似

#### 数据如何进入和离开

- 进入：`CreateWorkspaceRequest / UpdateWorkspaceRequest / WorkspaceQueryRequest`
- 中间：`WorkspaceEntity`
- 离开：`WorkspaceVO / PageResult<WorkspaceVO>`

#### 文件最容易看不懂的点

为什么工作空间模块没有复用 `ResourceAccessServiceImpl`？

因为它自己就是资源归属链的第一层，校验自己即可；`ResourceAccessServiceImpl` 更常用于 graph 和 graph 下的子资源。

### 3. `module/workspace/converter/WorkspaceConverter.java`

它的职责很纯：

- `WorkspaceEntity -> WorkspaceVO`

这是一个典型“样板但必要”的文件。  
它存在的意义不是做复杂逻辑，而是隔离数据库模型和返回模型。

## 四、样板文件怎么归类

### DTO

- `CreateWorkspaceRequest`
- `UpdateWorkspaceRequest`
- `WorkspaceQueryRequest`

作用：承接创建、更新、分页查询参数。

### VO

- `WorkspaceVO`

作用：前端看到的工作空间视图。

### Mapper / Entity

- `WorkspaceMapper`
- `WorkspaceEntity`

作用：数据库事实记录，不负责业务编排。

## 五、模块和其他模块的关系

这是所有 graph 的上游。

资源链是：

`current user -> workspace -> graph -> node/path/equipment/constraint`

所以你如果不理解工作空间，后面 graph 的权限链也会看得很散。
