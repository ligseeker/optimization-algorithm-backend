# 03_流程图核心模块

## 一、模块定位

流程图模块是整个平台的主聚合根。

它不只是“一个表的 CRUD”，而是负责：

1. 管理流程图基础信息
2. 聚合节点、路径、设备、约束形成 detail 视图
3. 通过 `graph_version` 配合缓存
4. 成为 YAML 导入导出和优化任务的数据源头

## 二、模块级三条线

### 请求流

```text
ApiGraphController
  -> GraphAppServiceImpl
  -> ResourceAccessServiceImpl
  -> FlowGraphMapper / NodeMapper / PathMapper / EquipmentMapper / ConstraintMapper
  -> GraphConverter / NodeConverter / PathConverter / EquipmentConverter / ConstraintConverter
  -> GraphVO / GraphDetailVO
```

### 数据流

```text
CreateGraphRequest / UpdateGraphRequest
  -> FlowGraphEntity
  -> GraphVO

graphId
  -> 查询 graph + nodes + paths + equipments + constraints
  -> GraphDetailVO
```

### 异常流

- 工作空间无权限 -> `FORBIDDEN`
- 流程图不存在 -> `RESOURCE_NOT_FOUND`
- 名称重复 -> `CONFLICT`
- 数据库不可用 -> `SYSTEM_ERROR`

## 三、关键文件逐个说明

### 1. `module/graph/controller/ApiGraphController.java`

#### 文件职责

graph 模块的 HTTP 入口。

#### 为什么会有这个文件

它把几个不同粒度的接口分开了：

- 工作空间下创建/列表
- graph 基础信息查询
- graph 聚合详情查询
- graph 更新/删除

#### 主要方法怎么看

- `createGraph`
  依附 `workspaceId`
- `listGraphs`
  依附 `workspaceId`
- `getGraph`
  只查 graph 本身
- `getGraphDetail`
  查 graph 的聚合视图
- `updateGraph`
  只改 graph 基础字段
- `deleteGraph`
  删除 graph 及其下属数据

#### 最容易看不懂的点

`getGraph` 和 `getGraphDetail` 是两种不同语义：

- `getGraph`：基础信息
- `getGraphDetail`：展示态聚合结果

### 2. `module/graph/service/impl/GraphAppServiceImpl.java`

这是 graph 模块最核心的文件。

#### 字段逐项说明

- 一组 `ObjectProvider<...Mapper>`
  说明这个类直接编排了 graph 及其多个子资源查询
- `resourceAccessService`
  先校验 graph / workspace 可访问
- `redisSafeClient`
  提供 detail 缓存
- `graphDetailTtlMinutes`
  控制 detail 缓存过期时间

#### 主要方法逐项说明

- `createGraph`
  1. 先校验 workspace 可访问
  2. 校验同 workspace 下 graph 名称唯一
  3. 组装 `FlowGraphEntity`
  4. 设置默认 `sourceType / graphStatus / graphVersion`
  5. 落库

- `listGraphs`
  只查某个 workspace 下的 graph 列表，支持关键字搜索和分页

- `getGraph`
  这是最薄的一种查询，只把 `FlowGraphEntity` 转成 `GraphVO`

- `getGraphDetail`
  这是本文件最重要的方法：
  1. 先拿可访问 graph
  2. 读取 `graphVersion`
  3. 组装缓存 key：`graph:detail:{graphId}:v{graphVersion}`
  4. 先查 Redis
  5. 未命中时，分别查 nodes / paths / equipments / constraints
  6. 组装 `GraphDetailVO`
  7. 回写缓存

- `updateGraph`
  更新 graph 自身基础字段

- `deleteGraph`
  按顺序清理约束、路径、节点、设备，再删 graph 本身

- `ensureGraphNameUnique`
  负责同工作空间下名称唯一约束

- `getXXXMapper`
  一组“数据库兜底入口”，数据库不可用时统一抛 `SYSTEM_ERROR`

#### 数据如何进入和离开

- 进入：`CreateGraphRequest / UpdateGraphRequest / GraphQueryRequest / graphId`
- 中间：`FlowGraphEntity + 多个子资源 Entity`
- 离开：`GraphVO / GraphDetailVO`

#### 最容易看不懂的点

这个文件为什么同时查这么多表？

因为它承担的是“聚合根”的职责，不只是 `flow_graph` 单表服务。

### 3. `module/graph/service/impl/GraphVersionServiceImpl.java`

#### 文件职责

专门负责 `graph_version` 自增。

#### 为什么单独拆出来

因为 node/path/equipment/constraint 的写操作都要碰它。  
如果把“版本号自增”散落在每个模块里，既重复又不统一。

#### 主要方法

- `increaseVersion`
  用 `setSql("graph_version = graph_version + 1")` 直接自增

#### 它在全局的意义

它不是业务功能点，而是缓存一致性的基础配套。

### 4. `module/graph/converter/GraphConverter.java`

#### 文件职责

把 `FlowGraphEntity` 转成 `GraphVO`

#### 为什么存在

它确保数据库模型不会直接暴露给前端，也让 graph 基础视图和 detail 聚合视图分层清楚。

## 四、样板文件怎么归类

### DTO

- `CreateGraphRequest`
- `UpdateGraphRequest`
- `GraphQueryRequest`

### VO

- `GraphVO`
- `GraphDetailVO`

注意：

- `GraphVO` 是 graph 自身视图
- `GraphDetailVO` 是 graph + 子资源的聚合视图

## 五、这个模块和其他模块的关系

- 上游依赖：`workspace`
- 下游依赖：`node / path / equipment / constraint`
- 横向依赖：
  - `ResourceAccessServiceImpl`
  - `RedisSafeClient`
  - `GraphVersionServiceImpl`
- 被下游复用：
  - `yaml`
  - `optimize`

## 六、读这个模块时最该抓住的认知

你如果只把 graph 看成“流程图表的 CRUD”，后面很多设计都会看不懂。

正确理解是：

- graph 是资源权限链的中枢
- graph 是 detail 聚合缓存的中心
- graph 是导入导出和优化任务的主数据源
