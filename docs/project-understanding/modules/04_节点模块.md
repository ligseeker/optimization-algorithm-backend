# 04_节点模块

## 一、模块定位

节点模块管理流程图中的节点记录。它表面上是 CRUD，实际上承担了 3 个额外职责：

1. 保证同 graph 下 `nodeCode` 唯一
2. 维护节点与设备的关联合法性
3. 删除节点时清理关联路径和约束

## 二、模块主链

```text
ApiNodeController
  -> NodeAppServiceImpl
  -> ResourceAccessServiceImpl
  -> ProcessNodeMapper / ProcessPathMapper / ConstraintMapper / EquipmentMapper
  -> GraphVersionServiceImpl
  -> NodeConverter
  -> NodeVO
```

## 三、关键文件

### 1. `module/node/service/impl/NodeAppServiceImpl.java`

这是节点模块真正的业务中心。

#### 字段逐项说明

- `processNodeMapperProvider`
  节点本表读写
- `processPathMapperProvider`
  删除节点时清理相关路径
- `constraintConditionMapperProvider`
  删除节点时清理相关约束
- `equipmentMapperProvider`
  校验设备是否属于当前 graph
- `resourceAccessService`
  统一校验 graph 可访问
- `graphVersionService`
  任意写操作后提升 graph 版本

#### 主要方法逐项说明

- `createNode`
  先校验 graph 权限，再校验 `nodeCode` 唯一，再校验设备归属，最后插入并自增 graph 版本

- `listNodes`
  基于 graph 查询，支持关键字和分页

- `getNode`
  本质是 `getNodeInGraph + converter`

- `updateNode`
  重新校验 `nodeCode` 唯一和设备归属，再更新

- `deleteNode`
  先删与该节点相关的 path 和 constraint，再删节点本身，然后自增 graph 版本

- `getNodeInGraph`
  节点级资源存在性校验入口

- `ensureNodeCodeUnique`
  负责同 graph 下业务唯一性

- `validateEquipmentIfPresent`
  防止节点引用其他 graph 的设备

#### 最容易看不懂的点

为什么删节点时还要碰 path 和 constraint？

因为节点是图结构的核心资源，删掉它后，依赖它的路径和约束会立即失效。

### 2. `module/node/converter/NodeConverter.java`

作用很纯：

- `ProcessNodeEntity -> NodeVO`

它本身不复杂，但它体现了“节点表结构”和“返回前端的节点视图”分离。

## 五、和其他模块的关系

- 上游依赖：`graph`
- 横向依赖：`equipment`
- 被这些模块引用：
  - `path`
  - `constraint`
  - `yaml`
  - `optimize`

所以节点模块虽然是子资源模块，但它是所有图结构资源里最核心的一层。
