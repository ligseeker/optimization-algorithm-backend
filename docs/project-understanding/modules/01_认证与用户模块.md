# 01_认证与用户模块

## 一、模块定位

这个模块解决的不是“资源权限”本身，而是更前置的两件事：

1. 你是谁
2. 你是不是已经登录

它的产出是后续所有业务模块都要依赖的“当前用户上下文”。

## 二、模块级请求流、数据流、异常流

### 请求流

```text
AuthController
  -> AuthServiceImpl / CurrentUserServiceImpl
  -> SysUserMapper / Sa-Token
  -> LoginResponseVO / CurrentUserVO
  -> Result<T>
```

### 数据流

```text
LoginRequest
  -> SysUserEntity
  -> AuthUserConverter
  -> LoginResponseVO
```

### 异常流

- 用户名或密码错误 -> `BusinessException(UNAUTHORIZED)`
- 用户被禁用 -> `BusinessException(FORBIDDEN)`
- 未登录访问当前用户接口 -> `StpUtil.checkLogin()` 抛 Sa-Token 异常
- 最终都由 `GlobalExceptionHandler` 统一转成 `Result<T>`

## 三、这个模块里最值得先看的文件

### 1. `module/auth/controller/AuthController.java`

#### 文件职责

认证接口入口，负责：

- 登录
- 退出
- 查询当前用户

#### 为什么会有这个文件

它把 HTTP 世界和业务世界隔开：

- HTTP 参数怎么接
- 返回结果怎么包
- 哪些接口要求登录

这些都留在 Controller 层。

#### 依赖了谁 / 被谁调用

- 依赖 `AuthService`
- 依赖 `CurrentUserService`
- 被前端直接调用

#### 文件里各部分怎么看

- `@RestController + @RequestMapping("/api/auth")`
  说明它是新 API 路径的一部分。
- `login`
  收 `LoginRequest`，返回 `LoginResponseVO`
- `logout`
  不做复杂逻辑，只转发给 `AuthService`
- `getCurrentUser`
  实际读取逻辑在 `CurrentUserService`

### 2. `module/auth/service/impl/AuthServiceImpl.java`

#### 文件职责

登录主流程都在这里。

#### 为什么会有这个文件

因为“登录”不是简单查一条用户记录，而是要串起：

- 查用户
- 校验密码
- 校验状态
- 创建登录态
- 更新最后登录时间
- 返回登录信息

#### 字段逐项说明

- `sysUserMapperProvider`
  用 `ObjectProvider` 延迟拿 Mapper，数据库不可用时可以手动抛业务异常，而不是直接注入失败。
- `currentUserService`
  用于 `getCurrentUser` 逻辑，不参与真正的登录校验。

#### 主要方法逐项说明

- `login(LoginRequest request)`
  这是登录主线：
  1. 取 `username`
  2. 查 `SysUserEntity`
  3. `PasswordUtils.matches` 校验密码哈希
  4. 校验 `status`
  5. `StpUtil.login(user.getId())`
  6. 更新 `lastLoginAt`
  7. 用 `AuthUserConverter` 组装 `LoginResponseVO`

- `logout()`
  先 `checkLogin`，再 `logout`

- `getCurrentUser()`
  不重复造逻辑，直接调用 `CurrentUserService`

- `getSysUserMapper()`
  这是一个典型的“数据库可用性兜底点”。

#### 数据如何进入、如何离开这个文件

- 进入：`LoginRequest`
- 中间：`SysUserEntity`
- 离开：`LoginResponseVO`

#### 可能抛出什么异常

- `UNAUTHORIZED`：用户不存在、密码不匹配
- `FORBIDDEN`：用户被禁用
- `SYSTEM_ERROR`：数据库 Mapper 不可用

#### 最容易看不懂的点

`login` 看起来很短，但其实它依赖了三个外部机制：

1. MyBatis 查用户
2. `PasswordUtils` 校验哈希
3. `StpUtil.login` 建立会话

### 3. `module/auth/service/impl/CurrentUserServiceImpl.java`

#### 文件职责

把“当前请求对应哪个用户”这件事统一收口。

#### 为什么会有这个文件

因为很多模块都要知道：

- 当前用户 ID
- 当前用户是不是 admin
- 当前用户是否仍然有效

如果每个模块自己做，会重复而且容易出错。

#### 字段逐项说明

- `sysUserMapperProvider`
  用来按登录态里的 userId 回查最新用户记录

#### 主要方法逐项说明

- `getCurrentUserId()`
  先 `StpUtil.checkLogin()`，再拿登录 ID

- `getCurrentUserEntity()`
  这是最关键的方法：
  1. 先拿当前 userId
  2. 再查数据库
  3. 校验用户是否仍存在
  4. 校验用户状态是否启用

- `isAdmin()`
  基于 `roleCode` 判断

#### 数据流

```text
Sa-Token loginId -> SysUserEntity -> 供其他模块复用
```

#### 异常流

- 未登录 -> Sa-Token 异常
- 用户被删 -> `UNAUTHORIZED`
- 用户被禁用 -> `FORBIDDEN`

### 4. `module/auth/support/SaTokenRoleProvider.java`

#### 文件职责

告诉 Sa-Token：某个登录用户拥有什么角色。

#### 为什么会有这个文件

Sa-Token 只管登录态，不知道数据库里的角色字段怎么取，所以需要一个 `StpInterface` 实现。

#### 字段逐项说明

- `sysUserPersistenceService`
  用持久化服务按 `loginId` 回查用户

#### 主要方法逐项说明

- `getPermissionList`
  当前项目没有做细粒度 permission，直接返回空列表

- `getRoleList`
  核心逻辑：
  1. 若 `loginId` 为空，返回空
  2. 查用户
  3. 取 `roleCode`
  4. 包成单元素列表返回给 Sa-Token

#### 最容易看不懂的点

它不是业务权限校验的终点。  
它只负责“角色来源”，资源级权限仍然要靠 `ResourceAccessServiceImpl`。

## 四、样板文件怎么理解

### DTO

- `LoginRequest`

作用：只承接登录参数。

### VO

- `CurrentUserVO`
- `LoginResponseVO`

作用：把用户基础信息和 token 返回给前端。

### Converter

- `AuthUserConverter`

作用：把 `SysUserEntity` 转成返回视图。

### 常量 / 工具

- `UserRoleCode`
- `PasswordUtils`

作用：

- `UserRoleCode` 统一角色码
- `PasswordUtils` 统一密码哈希和校验逻辑
