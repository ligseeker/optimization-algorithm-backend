# 10_操作日志模块

## 一、模块定位

操作日志模块是一个横切能力，不属于某个单独业务，但会附着在多个业务接口上。

它解决的问题是：

- 谁在什么时间做了什么操作
- 操作是否成功
- 如果失败，大致失败原因是什么

## 二、主链

```text
带 @OperationLog 的 Controller 方法
  -> OperationLogAspect.around
  -> 执行业务逻辑
  -> finally 中尝试落 OperationLogEntity
  -> OperationLogMapper.insert
```

## 三、关键文件

### 1. `module/log/annotation/OperationLog.java`

职责：

- 给接口打审计标记

主要信息：

- `operationType`
- `objectType`
- `objectIdParam`

它的意义是把“这个接口应该记录什么类型的操作”声明出来。

### 2. `module/log/aspect/OperationLogAspect.java`

这是日志模块的核心。

#### 字段逐项说明

- `operationLogMapperProvider`
  最终把日志写入数据库

#### 主要方法逐项说明

- `around`
  AOP 主入口：
  1. 记录开始时间
  2. 执行业务方法
  3. 捕获异常但不吞掉
  4. 在 `finally` 里写操作日志

- `persistOperationLog`
  真正组装 `OperationLogEntity`
  - 获取当前请求
  - 获取当前用户
  - 写入操作类型、对象类型、对象 ID、URI、耗时、成功标记、错误消息

- `resolveCurrentUserId`
  从 Sa-Token 登录态里尝试拿 userId

- `resolveObjectId`
  优先按注解指定的参数名拿对象 ID；如果拿不到，再尝试从 `Result.data` 里反射找 `id/taskId/graphId/workspaceId`

#### 最容易看不懂的点

这个切面即使业务抛异常，也会尽量记录日志，因为落日志在 `finally` 里。

### 3. `OperationLogEntity / OperationLogMapper`

作用：

- `OperationLogEntity` 是审计记录实体
- `OperationLogMapper` 负责落库

它们本身不复杂，复杂的是切面什么时候、从哪里拿哪些字段。

## 四、这个模块和其他模块的关系

它不是业务主链的一部分，但会附着在：

- 工作空间写操作
- graph 写操作
- YAML 导入导出
- 优化任务提交与查结果

所以它更像一个“围绕主链工作的观测点”。

## 五、读这个模块时最该抓住的认知

不要把它当“普通日志输出”。

它记录的不是 debug 信息，而是：

- 操作人
- 操作对象
- 操作结果
- 操作耗时

这是平台化以后很重要的审计能力。
