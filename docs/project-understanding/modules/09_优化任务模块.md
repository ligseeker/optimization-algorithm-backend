# 09_优化任务模块

## 一、模块定位

这是整个平台最复杂、也最能体现“平台化重构价值”的模块。

它负责：

1. 接收优化任务请求
2. 创建任务记录
3. 缓存任务状态
4. 事务提交后异步执行
5. 调算法
6. 落库结果
7. 更新状态
8. 支持失败重试

## 二、模块级三条线

### 请求流

```text
OptimizeTaskController
  -> OptimizeTaskAppServiceImpl
  -> ResourceAccessServiceImpl / CurrentUserServiceImpl
  -> OptimizeTaskMapper / OptimizeResultMapper
  -> OptimizeTaskStateServiceImpl / OptimizeTaskCacheServiceImpl
  -> AlgorithmExecutorImpl
  -> OptimizeTaskVO / OptimizeTaskSubmitVO / OptimizeResultVO
```

### 数据流

```text
CreateOptimizeTaskRequest
  -> OptimizeTaskEntity
  -> optimize_task
  -> AlgorithmExecutorImpl 构造 ProcessMap
  -> Algorithm1/2/3
  -> OptimizeResultEntity
  -> OptimizeResultVO
```

### 异常流

- graph 无权限 -> `FORBIDDEN`
- 任务不存在 -> `RESOURCE_NOT_FOUND`
- 任务未成功就查结果 -> `CONFLICT`
- 重试次数超限 -> `CONFLICT`
- 算法执行异常 -> 任务标记 `FAILED`
- 数据库不可用 -> `SYSTEM_ERROR`

## 三、关键文件逐个说明

### 1. `module/optimize/controller/OptimizeTaskController.java`

#### 文件职责

优化任务模块的 HTTP 入口。

#### 主要方法

- `submitTask`
- `listTasks`
- `getTask`
- `getTaskResult`
- `retryTask`

#### 最容易看不懂的点

Controller 本身很薄，但这里的每个接口都对应一条相对复杂的后端链路。

### 2. `module/optimize/service/impl/OptimizeTaskAppServiceImpl.java`

这是任务模块同步阶段的总编排器。

#### 字段逐项说明

- `optimizeTaskMapperProvider`
  任务表读写
- `optimizeResultMapperProvider`
  结果表查询
- `resourceAccessService`
  graph 权限校验
- `currentUserService`
  当前用户归属
- `algorithmExecutor`
  真正提交异步执行
- `objectMapper`
  结果反序列化时用
- `optimizeTaskCacheService`
  状态/结果缓存
- `optimizeTaskStateService`
  状态落库与缓存同步

#### 主要方法逐项说明

- `submitTask`
  主流程：
  1. 校验 graph 权限
  2. 获取当前用户
  3. 创建 `OptimizeTaskEntity`
  4. 初始状态设 `PENDING`
  5. 落库
  6. 缓存状态
  7. `submitAfterCommit`

- `listTasks`
  支持按 workspace / graph / 状态查询；普通用户只能看自己的任务

- `getTask`
  先查数据库，再尝试用 Redis 状态覆盖实时状态

- `getTaskResult`
  只有 `SUCCESS` 才允许查结果；优先读缓存，缓存未命中再回库

- `retryTask`
  核心点：
  1. 必须是 `FAILED`
  2. 必须没超过最大重试次数
  3. 先清理旧缓存
  4. 状态重置为 `PENDING`
  5. 事务提交后再次异步提交

- `submitAfterCommit`
  这是本文件最关键的方法之一：
  - 如果有事务，注册 `afterCommit`
  - 事务真正提交后再执行异步任务

#### 最容易看不懂的点

为什么不直接 `algorithmExecutor.submit(taskId)`？

因为如果事务还没提交，异步线程可能先跑起来，但数据库里任务记录还没真正可见。

### 3. `module/optimize/service/impl/OptimizeTaskStateServiceImpl.java`

#### 文件职责

统一维护任务状态的数据库更新和缓存同步。

#### 为什么单独拆出来

因为 `PENDING / RUNNING / SUCCESS / FAILED` 的更新逻辑会被多个地方调用，如果散在执行器和 AppService 里，会很乱。

#### 主要方法

- `markPending`
- `markRunning`
- `markSuccess`
- `markFailed`

每个方法都做两件事：

1. 更新数据库里的状态字段
2. 同步更新 Redis 状态缓存

### 4. `module/optimize/service/impl/OptimizeTaskCacheServiceImpl.java`

#### 文件职责

专门管理两类缓存：

- 任务状态缓存
- 优化结果缓存

#### 字段逐项说明

- `TASK_STATUS_KEY_PREFIX`
  `optimize:task:status:`
- `RESULT_KEY_PREFIX`
  `optimize:result:`
- TTL 配置字段
  控制状态和结果的缓存时间

#### 主要方法

- `cacheTaskStatus / getTaskStatus / evictTaskStatus`
- `cacheOptimizeResult / getOptimizeResult / evictOptimizeResult`

#### 最容易看不懂的点

这个类不做业务判断，它只是缓存适配层。  
“该不该缓存、何时删缓存”的决策不在这里，在 AppService 和 StateService。

### 5. `module/optimize/executor/impl/AlgorithmExecutorImpl.java`

这是整个项目里最复杂的文件之一。

#### 文件职责

真正把“任务记录”变成“算法执行”和“结果落库”。

#### 字段逐项说明

- 一组 `ObjectProvider<...Mapper>`
  说明执行阶段要自己去拿 graph 聚合数据
- `optimizeTaskExecutor`
  真正运行异步任务的线程池
- `objectMapper`
  结果 JSON 序列化
- `optimizeTaskStateService`
  更新任务状态
- `optimizeTaskCacheService`
  回写结果缓存

#### 主要方法逐项说明

- `submit`
  把任务塞进 `ThreadPoolTaskExecutor`

- `executeTask`
  异步任务统一入口：
  1. 查任务
  2. 只处理 `PENDING / RUNNING`
  3. 标记运行中
  4. 构造执行上下文
  5. 运行算法
  6. 成功则持久化结果
  7. 失败则标记失败

- `buildContext`
  查 graph、nodes、paths、equipments、constraints，再通过 `ProcessMapConverter` 组装算法输入

- `runAlgorithm`
  按 `algorithmType` 分发到 `Algorithm1/2/3`，并做：
  - diff 生成
  - mapCode 生成
  - 指标前后对比
  - scoreRatio 计算

- `persistSuccess`
  结果写入 `optimize_result`，再更新任务为成功，并缓存结果

- `persistFailure`
  统一失败出口，任务标记 `FAILED`

#### 内部辅助结构

- `OptimizeContext`
  把一次任务执行需要的 task、graph、sourceMap、factors 打包

- `AlgorithmOutput`
  把算法结果、diff、指标汇总成一份中间输出

#### 最容易看不懂的点

这个文件同时做了 4 件事：

1. 取平台数据
2. 组装算法输入
3. 调旧算法
4. 把算法输出重新包装成平台结果

所以它本质上是“平台层和算法层的总适配器”。

  - `Algorithm1/2/3`

## 六、读这个模块时最该抓住的认知

不要把它理解成“调用算法的一个接口”。

更准确的理解是：

它把一次算法运行，包装成了一个可查询、可重试、可缓存、可审计的后台任务。
