# 02_整体架构与核心调用链

## 一、整体分层先看懂

当前主路径可以简化成下面这张图：

```text
HTTP Request
  -> Controller
  -> AppService / 领域服务
  -> ResourceAccessService / CurrentUserService
  -> Mapper / Persistence / Redis / ThreadPool
  -> Converter / 算法适配层
  -> Result<T>
```

对应到目录：

### 1. 接口入口层

- `module/*/controller`
- 少量旧 `Controller`

职责：

- 接收 HTTP 请求
- 做注解校验
- 调业务服务
- 把结果包成 `Result<T>`

### 2. 业务编排层

- `module/*/service/impl/*AppServiceImpl`
- `module/common/service/impl/ResourceAccessServiceImpl`

职责：

- 组织一次完整业务流程
- 校验资源权限
- 决定事务边界
- 决定要不要走缓存、线程池、算法层

### 3. 持久化与基础设施层

- `infrastructure/persistence/entity`
- `infrastructure/persistence/mapper`
- `infrastructure/persistence/service`
- `common/cache/RedisSafeClient`
- `config/ThreadPoolConfig`

职责：

- Entity 对应表结构
- Mapper 直接和数据库交互
- RedisSafeClient 屏蔽 Redis 故障
- 线程池负责异步执行

### 4. 转换与适配层

- `module/*/converter`
- `module/yaml/converter/ProcessMapConverter`

职责：

- DTO -> Entity
- Entity -> VO
- 数据库模型 -> 算法模型
- 算法模型 -> 数据库模型

### 5. 统一异常出口

- `common/exception/BusinessException`
- `common/exception/ImportValidationException`
- `common/exception/GlobalExceptionHandler`

职责：

- 业务层只负责抛异常
- 最后统一转成 `Result<T>`

## 二、请求流转图

```text
前端请求
  -> @RestController
  -> 参数校验（@Valid / @Min / @ModelAttribute / @RequestBody）
  -> AppService
  -> ResourceAccessService / CurrentUserService
  -> Mapper / Redis / ThreadPool / Algorithm
  -> 返回 VO / PageResult / 布尔值
  -> Result.success(...)
  -> 前端收到统一响应
```

如果中途出错：

```text
参数错误 / 权限错误 / 业务错误 / 系统错误
  -> 抛异常
  -> GlobalExceptionHandler
  -> Result.fail(...)
  -> 前端收到统一错误码和消息
```

## 三、数据对象流转图

项目里最容易混的是“对象为什么这么多”。其实它们各有职责：

```text
请求参数
  DTO
    -> AppService 读取参数
    -> Entity 持久化
    -> Mapper 落库 / 查库
    -> Converter 转成 VO
    -> VO 返回前端

特殊链路：
  Entity 集合
    -> ProcessMapConverter
    -> ProcessMap / MultiNode / ProcessPath / ConstraintCondition / Equipment
    -> algorithm/*
```

### 这几类对象各自的角色

- DTO：只负责接收请求参数
- Entity：只负责数据库记录
- VO：只负责返回给前端
- 算法模型：只负责给旧算法包计算用
- Converter：负责“跨层翻译”

## 四、异常传播图

```text
参数注解校验失败
  -> MethodArgumentNotValidException / BindException / ConstraintViolationException
  -> GlobalExceptionHandler
  -> PARAM_INVALID

未登录 / 无角色
  -> Sa-Token 异常
  -> GlobalExceptionHandler
  -> UNAUTHORIZED / FORBIDDEN

业务规则不满足
  -> BusinessException
  -> GlobalExceptionHandler
  -> 对应 ErrorCode

YAML 导入校验失败
  -> ImportValidationException
  -> GlobalExceptionHandler
  -> FILE_PARSE_FAILED + ImportErrorReport

其他未预期异常
  -> Exception
  -> GlobalExceptionHandler
  -> SYSTEM_ERROR
```

## 五、三条最重要的调用链

### 1. 登录链

入口：

- `AuthController.login`

主流程：

1. 接收 `LoginRequest`
2. `AuthServiceImpl.login` 用 `SysUserMapper` 按用户名查用户
3. `PasswordUtils.matches` 校验密码哈希
4. 校验用户状态
5. `StpUtil.login(userId)` 建立登录态
6. 更新 `lastLoginAt`
7. `AuthUserConverter` 组装 `LoginResponseVO`
8. Controller 返回 `Result.success`

异常点：

- 用户不存在 / 密码错误 -> `UNAUTHORIZED`
- 用户禁用 -> `FORBIDDEN`
- 数据库不可用 -> `SYSTEM_ERROR`

### 2. graph detail 查询链

入口：

- `ApiGraphController.getGraphDetail`

主流程：

1. Controller 接收 `graphId`
2. `GraphAppServiceImpl.getGraphDetail`
3. `ResourceAccessServiceImpl.getAccessibleGraph` 校验资源可访问
4. 读取 `graph_version`
5. 组装缓存 key：`graph:detail:{graphId}:v{graphVersion}`
6. 先查 Redis
7. 未命中则分别查：
   - `flow_graph`
   - `process_node`
   - `process_path`
   - `equipment`
   - `constraint_condition`
8. 通过各类 Converter 组装 `GraphDetailVO`
9. 回写 Redis
10. Controller 返回 `Result<GraphDetailVO>`

异常点：

- graph 不存在 -> `RESOURCE_NOT_FOUND`
- graph 不属于当前用户 -> `FORBIDDEN`
- 数据库不可用 -> `SYSTEM_ERROR`

### 3. optimize task 提交与执行链

入口：

- `OptimizeTaskController.submitTask`

主流程分成两段。

#### 第一段：同步提交任务

1. 接收 `CreateOptimizeTaskRequest`
2. `OptimizeTaskAppServiceImpl.submitTask`
3. `ResourceAccessServiceImpl.getAccessibleGraph` 校验 graph 权限
4. `CurrentUserServiceImpl.getCurrentUserId` 获取当前用户
5. 创建 `OptimizeTaskEntity`
6. 初始状态设为 `PENDING`
7. 写入 `optimize_task`
8. `OptimizeTaskCacheServiceImpl` 缓存任务状态
9. 通过 `submitAfterCommit` 注册 `afterCommit`
10. 事务提交后调用 `AlgorithmExecutor.submit(taskId)`

#### 第二段：异步执行算法

1. `AlgorithmExecutorImpl.submit`
2. 交给 `optimizeTaskExecutor`
3. `executeTask`
4. `OptimizeTaskStateServiceImpl.markRunning`
5. 查询 graph、nodes、paths、equipments、constraints
6. `ProcessMapConverter.toProcessMap`
7. 转成算法需要的 `ProcessMap`
8. 按 `algorithmType` 调用 `Algorithm1/2/3`
9. 计算 diff、mapCode、指标前后值
10. 写入 `optimize_result`
11. `markSuccess` 更新任务状态
12. 缓存任务状态和结果

失败时：

1. 捕获异常
2. `persistFailure`
3. `markFailed`
4. 把错误码和错误消息写入任务记录

## 六、为什么这套结构难读，但其实是对的

因为它故意把职责拆散了：

- Controller 只保留外壳
- AppService 保留业务编排
- Mapper 只做数据访问
- Converter 只做对象转换
- ResourceAccessService 统一管资源权限
- GlobalExceptionHandler 统一管错误出口

所以你如果只打开一个文件，会觉得“这里怎么什么都没做完”；但从调用链角度看，每一层都很明确。
