# 06_事务边界与异常处理

## 一、先抓住一个总原则

这个项目的事务和异常设计，核心都在服务“平台化”：

- 写操作要尽量原子
- 失败要能定位清楚
- 异步任务要和同步事务边界解耦

## 二、哪些地方用了事务

典型入口：

- `AuthServiceImpl.login`
- `WorkspaceAppServiceImpl.create/update/delete`
- `GraphAppServiceImpl.create/update/delete`
- `NodeAppServiceImpl.create/update/delete`
- `PathAppServiceImpl.create/update/delete`
- `EquipmentAppServiceImpl.create/update/delete`
- `ConstraintAppServiceImpl.create/update/delete`
- `GraphYamlServiceImpl.importGraph`
- `OptimizeTaskAppServiceImpl.submitTask/retryTask`

这些地方的共同点是：

- 都是写操作
- 需要保证一组数据库动作尽量一致

## 三、最关键的事务边界：优化任务提交

重点文件：

- `OptimizeTaskAppServiceImpl`

为什么它最关键？

因为它同时碰到了：

- 数据库写入
- Redis 状态写入
- 异步线程启动

这里的关键设计是：

1. 先在事务里插入任务记录
2. 事务提交后再启动异步执行

对应代码：

- `submitAfterCommit`

这就是“同步事务”和“异步任务”之间的分界线。

## 四、YAML 导入为什么也要事务

重点文件：

- `GraphYamlServiceImpl.importGraph`

因为导入不是单表行为，而是：

1. 插 graph
2. 插 equipment
3. 插 nodes
4. 插 paths
5. 插 constraints

如果中途失败，不应该留下半套数据。

所以整个导入过程需要一个写事务包起来。

## 五、异常类型是怎么分层的

### 1. `BusinessException`

角色：

- 表达业务规则错误
- 带 `ErrorCode`

常见场景：

- 名称重复
- 无权限
- 资源不存在
- 任务状态不允许重试

### 2. `ImportValidationException`

角色：

- 专门服务 YAML 导入场景
- 附带 `ImportErrorReport`

它和普通 `BusinessException` 最大区别是：

- 不只是一个 message
- 还要把结构化校验错误详情带回前端

### 3. Sa-Token 异常

如：

- `NotLoginException`
- `NotRoleException`
- `NotPermissionException`

### 4. 其他系统异常

未预料的异常最终落到：

- `Exception.class` 处理分支

## 六、`GlobalExceptionHandler` 是怎么统一收口的

这是项目里所有异常对外返回的最后出口。

### 它做了什么

- 未登录 -> `UNAUTHORIZED`
- 无角色/无权限 -> `FORBIDDEN`
- `BusinessException` -> 保留业务错误码和 message
- `ImportValidationException` -> 返回 `ImportErrorReport`
- 参数校验异常 -> `PARAM_INVALID`
- `DuplicateKeyException` -> `CONFLICT`
- 其他异常 -> `SYSTEM_ERROR`

### 这意味着什么

业务层不需要自己拼响应 JSON。  
业务层只要“在正确的时候抛出正确类型的异常”。

## 七、参数校验异常是怎么进来的

常见来源：

- `@Valid`
- `@Min`
- `@ModelAttribute`
- `@RequestBody`

对应进入：

- `MethodArgumentNotValidException`
- `BindException`
- `ConstraintViolationException`
- `HttpMessageNotReadableException`

这些异常都不是业务层手写抛出的，而是 Spring MVC 参数绑定和校验阶段自动抛出的。

## 八、读事务和异常时最该抓住的点

1. 事务主要围绕写操作
2. 异步执行必须等事务提交后再启动
3. 业务代码负责抛异常，不负责拼统一响应
4. YAML 导入单独有一套结构化失败返回机制
