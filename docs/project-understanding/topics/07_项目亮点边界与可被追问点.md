# 07_项目亮点边界与可被追问点

## 一、可以放心讲的亮点

### 1. 渐进式重构

可以讲：

- 保留旧算法核心
- 新增平台层能力
- 通过转换层和执行器完成接入

这是当前项目最扎实的亮点之一。

### 2. 资源结构化建模

可以讲：

- 把原来文件里的流程图信息拆成 graph、node、path、equipment、constraint、task、result 等结构化资源

### 3. 异步任务化

可以讲：

- 把算法执行包装成可提交、可轮询、可重试、可查结果的任务

### 4. 权限收口

可以讲：

- 基于 Sa-Token 登录态 + 资源归属链做权限控制

### 5. 缓存一致性辅助设计

可以讲：

- 用 `graph_version` 配合 graph detail 缓存
- 任务状态与结果走 Redis 缓存

## 二、可以讲，但不要说太满的点

### 1. “架构设计”

可以说：

- 做了平台化分层与模块拆分

不建议说：

- 设计了复杂分布式架构

### 2. “高并发”

可以说：

- 用线程池支持异步任务执行

不建议说：

- 支撑高并发任务调度平台

### 3. “权限系统”

可以说：

- 做了角色 + 资源归属链访问控制

不建议说：

- 搭建了完整企业级权限中心

## 三、最容易被追问的点

### 1. 为什么不用 MQ

你需要能回答：

- 当前阶段为什么线程池足够
- 什么时候才值得上 MQ

### 2. 为什么不直接改算法模型

你需要能回答：

- 平台模型和算法模型的职责差异

### 3. `graph_version` 到底解决了什么

你需要能回答：

- 它是聚合缓存失效辅助机制，不是乐观锁设计重点

### 4. 失败重试怎么保证干净

你需要能回答：

- 清理旧状态缓存
- 清理旧结果缓存
- 重置任务状态

## 四、最稳妥的表达方式

最稳妥的说法是：

- “我基于现有算法包做了平台化重构”
- “我重点完成了资源结构化、权限收口、YAML 适配和异步任务执行链路”

这种说法既真实，也足够有技术含量。
