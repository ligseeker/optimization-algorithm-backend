# 01_权限模型与资源校验机制

## 一、先分清三件事

项目里的“权限”其实分 3 层：

1. 是否已登录
2. 是否拥有某个角色
3. 是否真的能访问某个资源

这三层分别由不同文件负责。

## 二、三层权限分别在哪里实现

### 1. 已登录

入口：

- `@SaCheckLogin`
- `StpUtil.checkLogin()`

作用：

- 先挡掉未登录请求

### 2. 角色来源

入口：

- `SaTokenRoleProvider`

作用：

- 告诉 Sa-Token 某个用户的 `roleCode`

当前项目里角色较简单，主要是：

- `ADMIN`
- `USER`

### 3. 资源访问权限

入口：

- `ResourceAccessServiceImpl`
- 个别模块自己的 `getAccessibleWorkspace`

作用：

- 就算你登录了，也未必能看别人的 workspace / graph

## 三、资源归属链

当前主链是：

```text
current user
  -> workspace
  -> graph
  -> node / path / equipment / constraint
```

这条链的核心含义是：

- `workspace` 是归属第一层
- `graph` 必须属于某个 workspace
- graph 下的所有子资源都默认继承 graph 的归属关系

## 四、`ResourceAccessServiceImpl` 为什么这么重要

这是整个项目资源权限的统一收口点。

### `getAccessibleWorkspace`

逻辑是：

1. 按 `workspaceId` 查 workspace
2. 不存在 -> `RESOURCE_NOT_FOUND`
3. 如果当前用户是 admin，直接放行
4. 否则要求 `workspace.ownerUserId == currentUserId`
5. 不满足 -> `FORBIDDEN`

### `getAccessibleGraph`

逻辑是：

1. 按 `graphId` 查 graph
2. graph 不存在 -> `RESOURCE_NOT_FOUND`
3. 再调用 `getAccessibleWorkspace(graph.workspaceId)`
4. 如果上游 workspace 无权限，则 graph 也无权限

### `checkGraphBelongsWorkspace`

逻辑是：

1. 先确保 graph 本身可访问
2. 再确认它属于指定 workspace
3. 不属于 -> `PARAM_INVALID`

## 五、为什么权限校验要做成这种“链式传递”

### 好处 1：避免每个模块重复写权限判断

graph、node、path、equipment、constraint、yaml、optimize 都会碰到 graph 归属问题。  
统一收口可以避免到处手写 owner 校验。

### 好处 2：不容易出现越权漏洞

如果只按 `graphId` 查，不回溯 workspace，就容易发生：

- 你知道 graphId
- 但这个 graph 不是你的
- 却仍然被查出来

### 好处 3：模块分工更清楚

- `CurrentUserServiceImpl` 只负责“当前用户是谁”
- `ResourceAccessServiceImpl` 只负责“当前用户能不能碰这个资源”

## 六、各模块里的权限落点

### workspace

在 `WorkspaceAppServiceImpl` 内部做校验，因为 workspace 自己就是第一层资源。

### graph

统一走 `ResourceAccessServiceImpl.getAccessibleWorkspace/getAccessibleGraph`

### node / path / equipment / constraint

先校验 graph 可访问，再校验本资源属于当前 graph

### yaml

- 导入：校验 workspace 可访问
- 导出：校验 graph 可访问

### optimize

- 提交任务：校验 graph 可访问
- 查询任务：先拿 task，再校验 task 对应 graph 可访问

## 七、读权限逻辑时最容易混淆的点

### 1. `SaCheckLogin` 不等于资源权限

它只能证明“你登录了”，不能证明“你有权访问这个 graph”。

### 2. admin 放行只发生在资源校验层

`CurrentUserServiceImpl.isAdmin()` 只是判断结果，真正“admin 可以看所有资源”的逻辑写在资源访问服务里。

### 3. 子资源权限不是单独建表实现的

当前项目没有为 node/path/equipment/constraint 单独做 owner 字段，而是通过 graph 归属链继承权限。
