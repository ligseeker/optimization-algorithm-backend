# 05_算法接入层与模型转换

## 一、这次重构最关键的技术问题是什么

不是“怎么写接口”，而是：

如何在不大改旧算法核心的前提下，把它接进一个新的平台架构。

这个问题的答案就是：

- 保留 `algorithm`
- 新增平台层
- 用转换层做桥接

## 二、两套模型世界

### 平台世界

- `FlowGraphEntity`
- `ProcessNodeEntity`
- `ProcessPathEntity`
- `EquipmentEntity`
- `ConstraintConditionEntity`

特点：

- 面向数据库
- 面向权限与查询
- 面向平台管理

### 算法世界

- `ProcessMap`
- `MultiNode`
- `ProcessPath`
- `ConstraintCondition`
- `Equipment`
- `InputInfo`

特点：

- 面向算法计算
- 面向图结构与约束求解

## 三、桥梁文件：`ProcessMapConverter`

这是整套设计里最重要的桥梁之一。

### 它解决了什么

#### 方向一：平台 -> 算法

graph + nodes + paths + equipments + constraints  
-> `ProcessMap`

#### 方向二：算法 / YAML -> 平台

`ProcessMap`
-> graph entity + node/path/equipment/constraint entity

## 四、为什么不能把算法模型直接拿来当 Entity

如果这么做，会有几个直接后果：

1. 数据库模型会跟着算法演进频繁波动
2. 权限、审计、分页、状态字段很难自然表达
3. 控制器和业务层会直接碰算法结构，耦合度过高

所以当前项目明确把平台模型和算法模型隔开。

## 五、真正调算法的是谁

不是 Controller，也不是 YAML Service。

真正的入口是：

- `AlgorithmExecutorImpl`

它会：

1. 从数据库查 graph 聚合数据
2. 用 `ProcessMapConverter.toProcessMap`
3. 根据 `algorithmType` 选择 `Algorithm1/2/3`
4. 拿到输出后再包装成平台可存储、可返回的结果

## 六、`algorithm/Main` 在新平台中的角色

`Main` 现在主要承担两类遗留能力：

1. 旧 YAML 解析/写出
2. 部分算法输出辅助逻辑

所以：

- `GraphYamlServiceImpl` 用它做导入导出桥接
- `AlgorithmExecutorImpl` 用它做 diff / mapCode 等辅助输出

## 七、读这块代码时最该抓住的认知

### 1. 新平台没有重写算法

而是重写了“算法外面的壳”。

### 2. `ProcessMapConverter` 是隔离带

它保护了两边：

- 平台层不必知道算法内部细节
- 算法层不必知道数据库和权限系统

### 3. `AlgorithmExecutorImpl` 是总适配器

它把平台世界和算法世界串成了一条真正可运行的生产链路。
