# 04_线程池异步任务与状态流转

## 一、为什么优化任务必须异步

算法执行时间不可控，如果同步做，会有几个问题：

1. HTTP 请求阻塞太久
2. 前端无法做任务状态轮询
3. 算法失败时难以保留中间状态和错误信息

所以项目把“优化计算”包装成后台任务。

## 二、异步主链长什么样

```text
OptimizeTaskController.submitTask
  -> OptimizeTaskAppServiceImpl.submitTask
  -> 插入 optimize_task，状态=PENDING
  -> afterCommit
  -> AlgorithmExecutorImpl.submit
  -> ThreadPoolTaskExecutor.execute
  -> executeTask
  -> markRunning
  -> buildContext
  -> runAlgorithm
  -> persistSuccess / persistFailure
```

## 三、线程池在哪儿配置

入口：

- `config/ThreadPoolConfig`

这里定义了：

- corePoolSize
- maxPoolSize
- queueCapacity
- keepAliveSeconds
- 线程名前缀 `optimize-task-`
- 拒绝策略 `CallerRunsPolicy`

## 四、为什么是 afterCommit，而不是直接提交线程池

关键代码在：

- `OptimizeTaskAppServiceImpl.submitAfterCommit`

意义是：

- 先保证任务记录已经成功提交到数据库
- 再让异步线程去查这个任务

如果跳过这一步，异步线程有可能在事务提交前就启动，导致：

- 查不到任务
- 查到旧状态
- 逻辑乱序

## 五、任务状态是怎么流转的

### 1. `PENDING`

来源：

- 新提交任务
- 失败后重试

### 2. `RUNNING`

来源：

- `AlgorithmExecutorImpl.executeTask`
- `OptimizeTaskStateServiceImpl.markRunning`

### 3. `SUCCESS`

来源：

- 算法执行成功
- 结果落库成功
- `markSuccess`

### 4. `FAILED`

来源：

- 算法报错
- 结果写库失败
- `persistFailure -> markFailed`

## 六、`OptimizeTaskStateServiceImpl` 的作用

它把状态迁移做成独立服务，统一负责：

1. 更新数据库里的任务状态字段
2. 同步更新 Redis 里的状态缓存

这样好处是：

- 执行器不需要自己手写所有状态 SQL
- 状态变更入口统一

## 七、为什么还要有 `OptimizeTaskCacheServiceImpl`

因为状态更新和缓存读写虽然相关，但职责不同：

- StateService：决定状态变成什么
- CacheService：决定状态和结果怎么读写缓存

## 八、失败重试为什么不能只是“再调一次算法”

重试前必须先做：

1. 校验当前状态必须是 `FAILED`
2. 校验重试次数没超限
3. 清理旧状态缓存
4. 清理旧结果缓存
5. 状态重置为 `PENDING`
6. 再次 `afterCommit` 提交线程池

否则会出现：

- 前端看到旧 SUCCESS/FAILED 缓存
- 新结果和旧结果混淆

## 九、为什么当前没用 MQ

当前项目选择线程池而不是 MQ，说明设计权衡是：

- 先满足异步化和状态化
- 控制系统复杂度
- 暂不引入消息中间件运维成本

这对当前阶段是合理的，但如果任务量继续放大，后面可能会演进到 MQ。
