# 05_面试问答总表

## 一、项目整体类

### 问：这个项目是做什么的？

答题主线：

- 原来是基于 YAML/JSON 文件的流程优化算法工具
- 重构后变成流程优化任务管理平台
- 平台不仅能跑算法，还能管理流程图资源、权限、任务状态和结果

### 问：为什么要重构？

答题主线：

- 文件方案不适合权限控制、审计、并发访问和任务追踪
- 需要把一次性算法运行变成可管理的后台任务
- 需要把流程图结构拆成数据库中的结构化资源

## 二、架构设计类

### 问：为什么会有这么多层？

答题主线：

- Controller 只做参数入口和统一返回
- AppService 负责业务编排
- Mapper / Entity 负责持久化
- Converter 负责跨层转换
- 这样做是为了隔离平台模型和算法模型

### 问：为什么旧目录和新目录共存？

答题主线：

- 这是渐进式重构，不是推倒重写
- 新结构承接平台化能力
- 旧 `algorithm` 保留核心算法
- 旧 `Controller / Service` 更多是历史遗留和过渡层

## 三、权限类

### 问：Sa-Token 负责了什么？

答题主线：

- Sa-Token 负责登录态
- `SaTokenRoleProvider` 负责角色来源
- 真正资源权限通过 `ResourceAccessServiceImpl` 落地

### 问：为什么要单独做资源访问服务？

答题主线：

- 避免每个模块重复写权限判断
- 防止 graph 和子资源越权访问
- 统一 workspace -> graph -> 子资源 的访问链

## 四、缓存类

### 问：Redis 缓存了什么？

答题主线：

- graph detail 聚合视图
- optimize task 状态
- optimize result 结果

### 问：为什么 `graph_version` 很关键？

答题主线：

- graph detail 是聚合缓存
- 任何子资源变化都应视为 detail 变化
- 用版本号切新 key，比局部更新简单且稳定

## 五、异步任务类

### 问：为什么不用 MQ，先用线程池？

答题主线：

- 当前阶段需求重点是异步化和状态化
- 线程池复杂度更低，足够承接当前任务规模
- 后续如果任务量更大，再演进到 MQ

### 问：为什么要 afterCommit 后再提交异步任务？

答题主线：

- 保证任务记录已经持久化成功
- 防止异步线程先跑起来却查不到任务

## 六、算法接入类

### 问：为什么不把 `ProcessMap` 直接改成数据库实体？

答题主线：

- 算法模型和平台模型职责不同
- 数据库需要权限、审计、分页、状态字段
- 直接混用会让平台层和算法层高度耦合

### 问：新平台怎么接旧算法？

答题主线：

- 通过 `ProcessMapConverter` 把数据库实体组装成算法模型
- `AlgorithmExecutorImpl` 统一选择 `Algorithm1/2/3`
- 把算法输出重新包装成平台结果

## 七、异常与事务类

### 问：异常是怎么统一处理的？

答题主线：

- 业务层抛 `BusinessException`
- 导入校验抛 `ImportValidationException`
- 参数校验和 Sa-Token 异常由 Spring/Sa-Token 抛出
- 最后都收口到 `GlobalExceptionHandler`

### 问：哪些操作需要事务？

答题主线：

- 所有写操作
- YAML 导入整链路
- optimize task 创建和重试入口
- 异步执行本身不包在同一个提交事务里
