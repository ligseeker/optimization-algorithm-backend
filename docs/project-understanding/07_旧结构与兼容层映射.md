# 07_旧结构与兼容层映射

## 为什么要单独写这篇

因为这个仓库不是“全新项目”，而是“渐进式重构项目”。

所以你会同时看到：

- 新结构：`module / infrastructure / common / config`
- 旧结构：`Controller / Service / algorithm`

如果不先讲清楚这层关系，你很容易在读代码时不停问：

- 为什么这里有两套 Controller
- 为什么这里又有一套 Service
- `algorithm` 到底是旧代码还是现役代码

## 一、新结构和旧结构各自代表什么

### 新结构：当前主路径

这部分是重构后的主体：

- `module`：业务模块
- `infrastructure/persistence`：数据库实体、Mapper、技术层服务
- `common`：统一返回、异常、缓存包装
- `config`：线程池、Redis、Sa-Token、MyBatis-Plus 等配置

理解重构后项目时，应该优先看这里。

### 旧结构：历史入口和过渡层

- `Controller`
- `Service`

这些目录更多反映重构前的组织方式。它们帮助你理解“项目从哪里来”，但不是你理解当前 `/api` 主路径的第一入口。

### `algorithm`：旧结构里最特殊的一块

它虽然在旧目录体系里，但并没有被废弃。

原因是：

- 核心优化算法仍然在这里
- 新平台没有重写算法，而是通过适配层接入

所以 `algorithm` 是“旧实现中的核心资产”，不是单纯历史垃圾。

## 二、旧目录和新目录的大致对应关系

### 旧 Controller -> 新 module/controller

旧目录中的这些入口：

- `AlgorithmController`
- `ConstraintController`
- `EquipmentController`
- `MapController`
- `NodeController`
- `PathController`

在重构后，大致被拆分到：

- `module/optimize/controller/OptimizeTaskController`
- `module/constraint/controller/ApiConstraintController`
- `module/equipment/controller/ApiEquipmentController`
- `module/graph/controller/ApiGraphController`
- `module/node/controller/ApiNodeController`
- `module/path/controller/ApiPathController`
- `module/yaml/controller/GraphYamlController`

区别在于：

- 旧 Controller 更像“按功能点堆接口”
- 新 Controller 更强调 REST 路径、统一 `/api` 前缀、统一 `Result<T>` 返回、统一鉴权和日志注解

### 旧 Service -> 新 AppService + 专项服务

旧目录中的：

- `AlgorithmService`
- `ConstraintService`
- `EquipmentService`
- `NodeService`
- `PathService`

在新结构下不再是“一层 Service 打天下”，而是被拆成：

- `*AppServiceImpl`：业务编排
- `ResourceAccessServiceImpl`：资源权限
- `GraphVersionServiceImpl`：版本号维护
- `OptimizeTaskStateServiceImpl`：任务状态
- `OptimizeTaskCacheServiceImpl`：缓存
- `AlgorithmExecutorImpl`：算法异步执行

这就是为什么你会觉得“现在文件变多了”。本质上不是功能变多，而是职责拆开了。

## 三、`algorithm` 包在新平台里的位置

### 1. `Main`

它现在主要承担两类桥接职责：

- 旧 YAML 解析/写出逻辑
- 路径 diff、mapCode 等算法侧输出能力

新平台里真正调用它的重点位置：

- `GraphYamlServiceImpl`
- `AlgorithmExecutorImpl`

### 2. `algorithm1 / algorithm2 / algorithm3`

它们仍然是实际优化算法的执行体。

新平台不会让 Controller 直接 `new Algorithm1/2/3`，而是统一通过：

- `OptimizeTaskAppServiceImpl` 提交任务
- `AlgorithmExecutorImpl` 在异步线程中调算法

### 3. `model/*`

这几类模型：

- `ProcessMap`
- `MultiNode`
- `ProcessPath`
- `ConstraintCondition`
- `Equipment`
- `InputInfo`

现在的角色是“算法领域模型”。

它们不等于：

- DTO
- Entity
- VO

它们只在“平台数据进入算法”或“算法结果回到平台”时出现，桥梁就是 `ProcessMapConverter`。

## 四、哪些旧文件现在还值得看

### 值得看

- `algorithm/Main.java`
- `algorithm/algorithm1/Algorithm1.java`
- `algorithm/algorithm2/Algorithm2.java`
- `algorithm/algorithm3/Algorithm3.java`
- `algorithm/model/*`

因为它们仍然参与当前运行链路。

### 一般作为历史参考

- 旧 `Controller`
- 旧 `Service`

它们更适合用来回答：

- 原来项目是怎么组织的
- 为什么后来要拆成现在这样

但如果你的目标是“理解重构后主路径”，优先级要低于新结构。

### 不需要逐个深挖

`algorithm` 目录下很多：

- `.yaml`
- `.csv`
- `.json`

这些多半是样例、测试输入或临时文件，不是当前平台架构理解重点。

## 五、正确的阅读切换方式

### 先看新结构，再回头看旧结构

推荐顺序：

1. 新 Controller
2. 新 AppService
3. 新权限/异常/缓存/异步核心
4. 再回头看旧目录对应物

### 不要把旧文件和新文件当成“一对一重命名”

很多时候并不是：

- 旧一个 Service
- 变成新一个 Service

而是：

- 旧一个大 Service
- 被拆成多个职责清晰的小服务

这就是为什么你在新结构里会看到更多文件，但每个文件更薄。
