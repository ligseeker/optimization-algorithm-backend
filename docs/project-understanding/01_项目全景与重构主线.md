# 01_项目全景与重构主线

## 一句话理解项目

这个项目原来更像“基于 YAML/JSON 文件的流程优化算法工具”，重构后更像“带权限、带持久化、带缓存、带异步任务调度的流程优化任务管理平台”。

## 重构前后最大的变化

### 重构前

重心在 `algorithm` 包：

- 输入主要是 YAML / JSON / 临时文件
- 算法直接围绕 `ProcessMap`、`MultiNode`、`ProcessPath` 等模型工作
- 更偏“单次运行算法”
- 资源、权限、审计、历史结果追踪能力较弱

### 重构后

重心移到平台层：

- Spring Boot 提供 API 入口
- MySQL 成为事实数据源
- Redis 负责热点缓存和任务状态缓存
- Sa-Token 负责登录态
- ThreadPoolTaskExecutor 负责异步优化任务
- YAML 被降级为导入导出格式

## 为什么会出现“新旧结构共存”

这是一个渐进式重构，不是推倒重来。

所以仓库里会同时看到两套世界：

### 新结构

- `module`
- `infrastructure/persistence`
- `common`
- `config`

这一套是当前主路径，新的 `/api` 接口、新的分层设计、新的任务平台能力都在这里。

### 旧结构

- `Controller`
- `Service`
- `algorithm`

这里面：

- `algorithm` 仍然很重要，因为核心算法还在用
- 旧 `Controller / Service` 更多是历史遗留或过渡产物，不再是主设计中心

## 当前系统的主业务闭环

从用户视角看，现在的系统核心闭环是：

1. 登录
2. 查询或创建工作空间
3. 在工作空间下维护流程图
4. 在流程图下维护节点、路径、设备、约束
5. 可选地导入/导出 YAML
6. 提交优化任务
7. 异步执行算法
8. 查询任务状态和优化结果
9. 记录关键操作日志

这个闭环说明项目已经从“算法工具”升级成“算法平台”。

## 平台化重构的核心设计主线

### 1. 资源结构化

原来很多信息散在 YAML 里，现在拆成了：

- `workspace`
- `flow_graph`
- `process_node`
- `process_path`
- `equipment`
- `constraint_condition`
- `optimize_task`
- `optimize_result`

这样做的意义是：

- 能做权限控制
- 能做事务
- 能做分页查询
- 能做缓存和日志
- 能追踪任务历史

### 2. 算法与平台解耦

`ProcessMap` 这类算法模型没有被直接改成数据库实体，而是通过 `ProcessMapConverter`、`AlgorithmExecutorImpl` 这样的适配层来衔接。

这保证了：

- 算法核心不被重构拖垮
- 平台层可以独立演进
- 导入导出和优化任务共享同一套转换逻辑

### 3. 权限和资源归属显式化

重构后不是“只要登录了就能调接口”，而是走清晰的资源归属链：

`当前用户 -> workspace -> graph -> 子资源`

这条链由 `CurrentUserServiceImpl`、`SaTokenRoleProvider` 和 `ResourceAccessServiceImpl` 共同支撑。

### 4. 同步请求与异步任务分离

优化计算不再堵在 HTTP 请求里，而是：

1. 先创建任务记录
2. 事务提交
3. 线程池异步执行
4. 写回状态和结果

这条主线是整个“任务平台化”的核心。

## 你读代码时最容易产生的错觉

### 错觉 1：`module` 里每个文件都在做业务

其实不是。很多文件只是：

- 接口参数承载
- 结果视图
- 类型转换
- 数据库访问封装

真正的业务编排主要集中在：

- `*AppServiceImpl`
- `ResourceAccessServiceImpl`
- `AlgorithmExecutorImpl`
- `GlobalExceptionHandler`

### 错觉 2：`algorithm` 包已经过时了

也不是。它不再是“系统入口”，但仍然是“优化计算引擎”。

### 错觉 3：Controller 里看不懂，说明业务逻辑分散

恰恰相反。重构后 Controller 故意变薄，真正逻辑被挪到了：

- AppService
- ResourceAccessService
- Converter
- Cache / Executor / ExceptionHandler
